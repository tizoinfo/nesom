# NESOM 系统开发需求文档

## 介绍

NESOM（新能源运维管理系统）是一个面向多品牌新能源设备的开放、中立运维管理平台。本文档基于已完成的概要设计和详细设计文档，定义系统开发的需求规格。

系统采用前后端分离架构：
- **前端**：Vue 3 + TypeScript + Element Plus
- **后端**：Python + FastAPI + SQLAlchemy
- **数据库**：MySQL 8.0 + Redis 7.0
- **文件存储**：MinIO
- **容器化**：Docker + Docker Compose

## 词汇表

- **NESOM**: 新能源运维管理系统
- **设备监控模块**: 负责设备状态实时监控、数据采集和告警管理
- **工单管理模块**: 负责维修、巡检、保养工单的全生命周期管理
- **用户权限管理模块**: 负责用户认证、RBAC权限控制和审计日志
- **备件管理模块**: 负责备件库存、入库出库和盘点管理
- **巡检管理模块**: 负责巡检计划、任务分配和现场执行
- **报表统计模块**: 负责数据统计、可视化和报表生成
- **系统配置模块**: 负责系统参数、字典数据和通知模板管理
- **JWT**: JSON Web Token，用于用户认证
- **RBAC**: 基于角色的访问控制
- **WebSocket**: 用于实时数据推送的双向通信协议

## 需求

### 需求 1：项目基础框架搭建

**用户故事**：作为开发团队，我希望有一个完整的项目基础框架，以便能够快速开始各模块的开发工作。

#### 验收标准

1. THE 后端框架 SHALL 使用 FastAPI 0.104+ 和 Python 3.11+，包含完整的项目目录结构
2. THE 前端框架 SHALL 使用 Vue 3.4 + TypeScript 5.3 + Element Plus 2.7，包含完整的项目目录结构
3. WHEN Docker Compose 启动时，THE 系统 SHALL 能够正常启动 MySQL、Redis、MinIO 和后端服务
4. THE 后端 SHALL 提供 /health 健康检查端点，返回服务状态
5. THE 前端 SHALL 能够通过 Vite 开发服务器正常运行
6. THE 数据库 SHALL 通过 Alembic 迁移脚本自动创建所有表结构

### 需求 2：用户权限管理模块

**用户故事**：作为系统管理员，我希望能够管理用户账户和权限，以便控制系统访问。

#### 验收标准

1. WHEN 用户提交正确的用户名和密码时，THE 认证服务 SHALL 返回 JWT 访问令牌和刷新令牌
2. WHEN 用户提交错误密码连续 5 次时，THE 认证服务 SHALL 锁定账户 30 分钟
3. THE 系统 SHALL 支持 RBAC 权限模型，包含管理员、工程师、操作员、观察员四种角色
4. WHEN JWT 令牌过期时，THE 系统 SHALL 允许使用刷新令牌获取新的访问令牌
5. THE 系统 SHALL 记录所有关键操作的审计日志，保留 180 天
6. THE 用户管理 API SHALL 支持用户的增删改查操作
7. THE 角色管理 API SHALL 支持角色和权限的配置

### 需求 3：设备监控模块

**用户故事**：作为运维人员，我希望能够实时监控设备状态，以便及时发现和处理设备问题。

#### 验收标准

1. THE 设备管理 API SHALL 支持设备的增删改查操作
2. WHEN 设备心跳超过 5 分钟未更新时，THE 系统 SHALL 自动将设备状态标记为离线
3. THE 系统 SHALL 通过 WebSocket 实时推送设备状态变更和告警信息
4. WHEN 设备指标超过阈值时，THE 告警服务 SHALL 生成告警记录并发送通知
5. THE 历史数据查询 API SHALL 支持按时间范围、聚合方式查询设备指标数据
6. THE 设备监控前端 SHALL 展示实时设备状态看板，包含状态统计和趋势图表

### 需求 4：工单管理模块

**用户故事**：作为调度员，我希望能够创建和管理工单，以便协调维修和保养工作。

#### 验收标准

1. THE 工单管理 API SHALL 支持工单的创建、分配、处理、审核、关闭全流程
2. THE 工单状态机 SHALL 按照设计文档定义的状态流转规则执行
3. WHEN 工单超过处理时限时，THE 系统 SHALL 自动升级工单优先级并发送通知
4. THE 工单 API SHALL 支持从工单模板快速创建标准化工单
5. THE 工单前端 SHALL 支持 PC 端表格视图和移动端卡片视图
6. THE 工单统计 API SHALL 提供工单数量、处理时效、人员绩效等统计数据

### 需求 5：备件管理模块

**用户故事**：作为仓库管理员，我希望能够管理备件库存，以便支持设备维修工作。

#### 验收标准

1. THE 备件管理 API SHALL 支持备件档案的增删改查操作
2. THE 库存管理 API SHALL 支持入库、出库、调拨、盘点操作
3. WHEN 备件库存低于安全库存时，THE 系统 SHALL 生成库存预警通知
4. THE 库存操作 SHALL 使用数据库事务保证数据一致性，防止超卖
5. THE 备件管理前端 SHALL 支持扫码出入库操作

### 需求 6：巡检管理模块

**用户故事**：作为巡检人员，我希望能够按照计划执行设备巡检，以便及时发现设备隐患。

#### 验收标准

1. THE 巡检计划 API SHALL 支持周期性巡检计划的创建和管理
2. THE 巡检任务 API SHALL 支持任务的自动生成、分配和执行
3. THE 移动端巡检 API SHALL 支持离线数据采集和同步
4. WHEN 巡检发现异常时，THE 系统 SHALL 支持自动创建关联工单
5. THE 巡检报告 API SHALL 支持自动生成巡检统计报告

### 需求 7：报表统计模块

**用户故事**：作为管理人员，我希望能够查看系统运营数据报表，以便做出决策。

#### 验收标准

1. THE 仪表盘 API SHALL 提供设备在线率、工单完成率、巡检完成率等 KPI 数据
2. THE 报表 API SHALL 支持设备运行统计、工单统计、备件使用统计等多维度报表
3. THE 报表导出 API SHALL 支持 Excel 和 PDF 格式导出
4. WHEN 报表数据量较大时，THE 系统 SHALL 使用异步任务生成报表
5. THE 统计数据 SHALL 使用 Redis 缓存，减少数据库查询压力

### 需求 8：系统配置模块

**用户故事**：作为系统管理员，我希望能够配置系统参数，以便根据业务需求调整系统行为。

#### 验收标准

1. THE 系统配置 API SHALL 支持系统参数的查询和更新
2. THE 字典管理 API SHALL 支持下拉选项数据的管理
3. THE 通知模板 API SHALL 支持邮件、短信通知模板的配置
4. WHEN 系统配置更新时，THE 系统 SHALL 自动刷新相关缓存
5. THE 系统健康检查 API SHALL 返回数据库、Redis、MinIO 等组件的连接状态
