## 1. 设计概述

### 1.1 业务领域
报表统计模块是NESOM系统的数据分析核心，负责多维度数据统计、可视化展示和决策支持。主要业务领域包括：

1. **报表管理**：模板定义、参数配置、拖拽式设计
2. **数据查询**：多源数据集成、参数化查询、缓存优化
3. **统计分析**：设备运行统计、能源生产分析、维护成本计算
4. **可视化展示**：图表渲染、仪表盘配置、实时刷新
5. **任务调度**：定时报表生成、异步任务处理、进度跟踪

### 1.2 核心业务流程
1. **报表生成流程**：模板选择 → 参数设置 → 查询执行 → 数据处理 → 可视化渲染 → 导出分发
2. **统计计算流程**：数据抽取 → 维度聚合 → 指标计算 → 结果缓存 → 前端展示
3. **仪表盘流程**：布局设计 → 组件配置 → 数据绑定 → 实时刷新 → 用户交互
4. **异步任务流程**：任务创建 → 队列分配 → 执行监控 → 结果存储 → 用户通知

### 1.3 设计原则
1. **性能优先**：大数据量查询优化，多级缓存，异步处理
2. **灵活性**：可配置报表模板，插件式图表，自定义计算
3. **安全性**：行级数据权限，SQL注入防护，操作审计
4. **用户体验**：响应式界面，实时数据，个性化配置
5. **可扩展性**：支持新数据源，新图表类型，新计算指标

## 2. 核心业务实体状态机

### 2.1 报表执行状态机 (Report Execution State Machine)

#### 2.1.1 状态定义
| 状态 | 编码 | 描述 | 业务含义 |
|------|------|------|----------|
| 待处理 | pending | 任务已创建，等待执行 | 任务在队列中排队 |
| 执行中 | running | 任务正在执行 | 数据查询或处理中 |
| 已完成 | completed | 任务执行成功 | 结果可用，可下载或查看 |
| 已失败 | failed | 任务执行失败 | 执行过程中出现错误 |
| 已取消 | cancelled | 任务被用户取消 | 用户主动取消执行 |
| 部分完成 | partial | 部分数据可用 | 大数据量分块处理中 |

#### 2.1.2 状态转换规则
```
                            ┌─────────────┐
                            │   pending   │◀────────────────────────┐
                            └─────────────┘                         │
                                   │                                │
         任务调度执行               │ 用户取消任务                    │
                                   ▼                                │
                            ┌─────────────┐                         │
          ┌─────────────────│  running    │─────────────────┐       │
          │                 └─────────────┘                 │       │
          │                        │                        │       │
          │执行成功                │ 执行失败                │       │
          ▼                        ▼                        ▼       │
    ┌─────────────┐        ┌─────────────┐        ┌─────────────┐   │
    │ completed   │        │   failed    │        │  cancelled  │   │
    └─────────────┘        └─────────────┘        └─────────────┘   │
          │                        │                        │       │
          │结果清理/重新执行        │ 重试/放弃              │       │
          └────────────────────────┴────────────────────────┘       │
                                                                   │
                                           用户重新提交任务         │
                                                                   │
                                            └───────────────────────┘
```

#### 2.1.3 状态转换约束表
| 当前状态 | 目标状态 | 触发条件 | 前置检查 | 后置动作 |
|----------|----------|----------|----------|----------|
| pending | running | 1. 任务调度器选取<br>2. 手动立即执行 | 1. 模板有效<br>2. 参数验证通过<br>3. 用户有权限 | 1. 记录开始时间<br>2. 更新任务状态<br>3. 开始进度跟踪 |
| running | completed | 1. 查询执行成功<br>2. 数据处理完成 | 1. 数据验证通过<br>2. 结果存储成功 | 1. 记录结束时间<br>2. 计算执行耗时<br>3. 生成结果文件<br>4. 发送完成通知 |
| running | failed | 1. 查询超时<br>2. 数据源错误<br>3. 内存不足 | 1. 错误重试次数已满<br>2. 错误不可恢复 | 1. 记录错误信息<br>2. 清理临时资源<br>3. 发送失败通知<br>4. 创建错误报告 |
| running | partial | 1. 大数据量分块处理<br>2. 流式处理中 | 1. 支持分块处理<br>2. 用户允许部分结果 | 1. 存储已处理数据<br>2. 更新进度信息<br>3. 继续处理下一块 |
| pending | cancelled | 1. 用户取消任务<br>2. 计划任务被禁用 | 1. 任务可取消<br>2. 未开始执行 | 1. 更新任务状态<br>2. 清理队列位置<br>3. 发送取消通知 |
| running | cancelled | 1. 用户强制取消<br>2. 系统资源不足 | 1. 任务支持取消<br>2. 安全停止检查 | 1. 终止执行进程<br>2. 清理临时资源<br>3. 记录取消原因 |
| failed | pending | 1. 用户重试任务<br>2. 自动重试机制 | 1. 最大重试次数未满<br>2. 错误原因已修复 | 1. 重置任务状态<br>2. 重新加入队列<br>3. 增加重试计数 |

#### 2.1.4 状态转换业务规则
1. **自动转换规则**：
   - 执行超时（>30分钟）：running → failed（自动）
   - 队列等待超时（>24小时）：pending → cancelled（自动）
   - 计划任务禁用：pending → cancelled（自动）

2. **人工干预规则**：
   - failed → pending：需分析错误原因，手动重试
   - running → cancelled：需确认数据一致性
   - partial → completed：需确认所有数据块处理完成

#### 3.1.1 动态SQL构建和执行
**输入**: 报表模板配置 + 用户参数
**输出**: 结构化数据结果
**处理流程**:

```python
def execute_report(template, parameters):
    # 1. 参数验证和安全检查
    validate_parameters(template.parameter_definitions, parameters)
    check_permissions(template, parameters)
    
    # 2. 构建参数化查询
    sql_template = template.data_source_config.query
    sql_params = build_sql_parameters(parameters)
    
    # 3. 安全检查：防止SQL注入
    safe_sql = sanitize_sql(sql_template, sql_params)
    
    # 4. 查询执行（支持缓存）
    cache_key = generate_cache_key(template.id, parameters)
    if cache_enabled and cache.exists(cache_key):
        return cache.get(cache_key)
    
    # 5. 数据库查询
    result = execute_safe_query(safe_sql, sql_params)
    
    # 6. 数据处理和格式化
    processed_data = process_result(result, template.column_definitions)
    
    # 7. 缓存结果
    if cache_enabled:
        cache.set(cache_key, processed_data, ttl=template.cache_ttl)
    
    return processed_data
```

**安全措施**:
1. **参数化查询**: 所有用户输入作为参数传递，不直接拼接SQL
2. **查询白名单**: 只允许预定义的查询模板
3. **复杂度限制**: 限制查询执行时间、返回行数、内存使用
4. **权限过滤**: 自动添加行级权限过滤条件

### 3.2 数据聚合逻辑

#### 3.2.1 时间维度聚合
**支持的时间粒度**:
- 小时 (hourly)
- 日 (daily)
- 周 (weekly)
- 月 (monthly)
- 季度 (quarterly)
- 年 (yearly)

**聚合算法**:
```python
def aggregate_by_time(data, time_field, value_field, aggregation_type):
    """
    按时间维度聚合数据
    
    Args:
        data: 原始数据列表
        time_field: 时间字段名
        value_field: 数值字段名
        aggregation_type: 聚合类型(sum/avg/min/max/count)
    
    Returns:
        按时间分组聚合后的数据
    """
    grouped = {}
    
    for record in data:
        # 提取时间并标准化到指定粒度
        timestamp = record[time_field]
        time_key = normalize_time(timestamp, granularity)
        
        if time_key not in grouped:
            grouped[time_key] = {
                'count': 0,
                'values': [],
                'sum': 0
            }
        
        value = record[value_field]
        grouped[time_key]['values'].append(value)
        grouped[time_key]['sum'] += value
        grouped[time_key]['count'] += 1
    
    # 计算聚合结果
    result = []
    for time_key, stats in grouped.items():
        if aggregation_type == 'sum':
            agg_value = stats['sum']
        elif aggregation_type == 'avg':
            agg_value = stats['sum'] / stats['count']
        elif aggregation_type == 'min':
            agg_value = min(stats['values'])
        elif aggregation_type == 'max':
            agg_value = max(stats['values'])
        elif aggregation_type == 'count':
            agg_value = stats['count']
        
        result.append({
            'time': time_key,
            'value': agg_value
        })
    
    return sorted(result, key=lambda x: x['time'])
```

#### 3.2.2 设备维度聚合
**聚合维度**:
- 场站 (station)
- 设备类型 (device_type)
- 制造商 (manufacturer)
- 型号 (model)
- 区域 (area)

**多维度交叉聚合**:
```python
def multi_dimension_aggregate(data, dimensions, metrics):
    """
    多维度交叉聚合
    
    Args:
        data: 原始数据
        dimensions: 维度列表 ['station', 'device_type']
        metrics: 指标配置 [{'field': 'availability', 'type': 'avg'}]
    
    Returns:
        多维透视表数据
    """
    # 构建多级分组键
    grouped = defaultdict(lambda: defaultdict(list))
    
    for record in data:
        # 生成复合分组键
        group_key = tuple(record[dim] for dim in dimensions)
        
        # 收集指标值
        for metric in metrics:
            field = metric['field']
            if field in record:
                grouped[group_key][field].append(record[field])
    
    # 计算聚合结果
    result = []
    for group_key, metrics_data in grouped.items():
        row = {}
        
        # 添加维度值
        for i, dim in enumerate(dimensions):
            row[dim] = group_key[i]
        
        # 计算指标值
        for metric in metrics:
            field = metric['field']
            agg_type = metric.get('type', 'avg')
            values = metrics_data.get(field, [])
            
            if agg_type == 'avg' and values:
                row[f"{field}_{agg_type}"] = sum(values) / len(values)
            elif agg_type == 'sum' and values:
                row[f"{field}_{agg_type}"] = sum(values)
            elif agg_type == 'count':
                row[f"{field}_{agg_type}"] = len(values)
            else:
                row[f"{field}_{agg_type}"] = None
        
        result.append(row)
    
    return result
```

#### 3.2.3 性能指标计算
**核心KPI计算公式**:

1. **设备可用率 (Availability)**
```
Availability = (Total Time - Downtime) / Total Time × 100%
```

2. **性能率 (Performance)**
```
Performance = (Actual Output / Theoretical Maximum Output) × 100%
```

3. **质量率 (Quality)**
```
Quality = (Good Output / Total Output) × 100%
```

4. **整体设备效率 (OEE)**
```
OEE = Availability × Performance × Quality
```

5. **平均修复时间 (MTTR)**
```
MTTR = Total Repair Time / Number of Repairs
```

6. **平均故障间隔时间 (MTBF)**
```
MTBF = Total Operating Time / Number of Failures
```

**计算实现**:
```python
def calculate_kpi(device_data, time_period):
    """
    计算设备KPI指标
    
    Args:
        device_data: 设备运行数据
        time_period: 时间周期配置
    
    Returns:
        KPI计算结果
    """
    # 时间范围过滤
    filtered_data = filter_by_time(device_data, time_period)
    
    # 计算基础指标
    total_time = calculate_total_time(filtered_data, time_period)
    downtime = calculate_downtime(filtered_data)
    actual_output = calculate_actual_output(filtered_data)
    theoretical_output = calculate_theoretical_output(filtered_data)
    good_output = calculate_good_output(filtered_data)
    
    # 计算KPI
    availability = ((total_time - downtime) / total_time) * 100 if total_time > 0 else 0
    performance = (actual_output / theoretical_output) * 100 if theoretical_output > 0 else 0
    quality = (good_output / actual_output) * 100 if actual_output > 0 else 0
    oee = availability * performance * quality / 10000  # 转换为百分比
    
    return {
        'availability': round(availability, 2),
        'performance': round(performance, 2),
        'quality': round(quality, 2),
        'oee': round(oee, 2),
        'total_time': total_time,
        'downtime': downtime,
        'actual_output': actual_output
    }
```

### 3.3 缓存策略

#### 3.3.1 多级缓存架构
```
用户请求
    ↓
第一层：内存缓存 (Redis)
    ├── 命中 → 返回结果
    └── 未命中
          ↓
第二层：数据库缓存 (statistics_cache表)
    ├── 命中 → 返回结果 + 刷新Redis
    └── 未命中
          ↓
第三层：原始数据查询
          ↓
         计算 + 缓存结果 (同时更新Redis和数据库)
```

#### 3.3.2 缓存键设计
```python
def generate_cache_key(template_id, parameters, options=None):
    """
    生成缓存键
    
    格式: report:{template_id}:{md5_hash}
    """
    # 参数序列化和排序（确保相同参数生成相同键）
    param_str = json.dumps(parameters, sort_keys=True)
    
    # 包含选项影响缓存
    if options:
        option_str = json.dumps({
            'page': options.get('page'),
            'page_size': options.get('page_size'),
            'format': options.get('format')
        }, sort_keys=True)
        param_str += option_str
    
    # 生成MD5哈希
    hash_obj = hashlib.md5(param_str.encode('utf-8'))
    hash_digest = hash_obj.hexdigest()
    
    return f"report:{template_id}:{hash_digest}"
```

#### 3.3.3 缓存失效策略
1. **基于TTL的过期**:
   - 热点数据: 5分钟

   - 历史数据: 24小时

2. **基于数据变更的失效**:

def invalidate_cache_on_data_change(data_source, changed_ids):

3. **主动刷新机制**:

#### 3.4.2 报表生成任务
```python
# tasks/report_tasks.py
from celery import current_task
from celery.utils.log import get_task_logger

logger = get_task_logger(__name__)

@app.task(bind=True, max_retries=3, default_retry_delay=60)
def generate_large_report(self, template_id, parameters, options):
    """
    异步生成大型报表
    
    Args:
        template_id: 报表模板ID
        parameters: 查询参数
        options: 生成选项
    
    Returns:
        报表文件路径和元数据
    """
    try:
        # 更新任务状态
        self.update_state(state='PROGRESS',
                          meta={'progress': 0, 'message': '开始处理'})
        
        # 1. 验证模板和参数
        template = get_template(template_id)
        validate_parameters(template, parameters)
        
        # 2. 执行查询（分块处理）
        self.update_state(state='PROGRESS',
                          meta={'progress': 20, 'message': '执行数据查询'})
        
        data_chunks = execute_query_in_chunks(template, parameters, chunk_size=10000)
        total_chunks = estimate_chunk_count(template, parameters)
        
        # 3. 处理数据块
        processed_data = []
        for i, chunk in enumerate(data_chunks):
            # 处理当前数据块
            processed_chunk = process_data_chunk(chunk, template)
            processed_data.extend(processed_chunk)
            
            # 更新进度
            progress = 20 + (i / total_chunks) * 60
            self.update_state(state='PROGRESS',
                              meta={'progress': int(progress),
                                    'message': f'处理数据块 {i+1}/{total_chunks}'})
        
        # 4. 生成导出文件
        self.update_state(state='PROGRESS',
                          meta={'progress': 85, 'message': '生成导出文件'})
        
        export_format = options.get('format', 'excel')
        file_path = export_report(processed_data, template, export_format, options)
        
        # 5. 完成任务
        self.update_state(state='PROGRESS',
                          meta={'progress': 100, 'message': '任务完成'})
        
        return {
            'status': 'success',
            'file_path': file_path,
            'file_size': os.path.getsize(file_path),
            'row_count': len(processed_data)
        }
        
    except Exception as exc:
        logger.error(f'报表生成失败: {exc}')
        self.update_state(state='FAILURE',
                          meta={'progress': 0, 'message': str(exc)})
        raise self.retry(exc=exc)
```

### 4.1 事务管理
**报表生成事务**:
```python
@contextmanager
def report_generation_transaction(template_id, execution_id):
    """
    报表生成事务管理
    
    确保数据一致性：执行记录、结果文件、缓存状态
    """
    try:
        # 开始事务
        db.session.begin()
        
        # 创建执行记录
        execution = ReportExecution(
            id=execution_id,
            template_id=template_id,
            status='running',
            start_time=datetime.utcnow()
        )
        db.session.add(execution)
        db.session.flush()
        
        yield execution
        
        # 提交事务
        db.session.commit()
        
    except Exception as e:
        # 回滚事务
        db.session.rollback()
        
        # 更新执行记录状态
        if 'execution' in locals():
            execution.status = 'failed'
            execution.error_message = str(e)
            execution.end_time = datetime.utcnow()
            
            # 使用新会话保存失败状态
            with new_db_session() as session:
                session.add(execution)
                session.commit()
        
        raise
```

### 4.2 数据清理策略
**定期清理任务**:
```python
def cleanup_report_data():
    """
    清理过期报表数据
    
    清理策略:
    1. 成功执行记录保留90天
    2. 失败执行记录保留30天
    3. 临时文件保留24小时
    4. 过期缓存自动清理
    """
    # 1. 清理执行记录
    success_cutoff = datetime.utcnow() - timedelta(days=90)
    failed_cutoff = datetime.utcnow() - timedelta(days=30)
    
    # 删除过期成功记录
    db.session.query(ReportExecution)\
        .filter(ReportExecution.status == 'completed',
                ReportExecution.end_time < success_cutoff)\
        .delete(synchronize_session=False)
    
    # 删除过期失败记录
    db.session.query(ReportExecution)\
        .filter(ReportExecution.status == 'failed',
                ReportExecution.end_time < failed_cutoff)\
        .delete(synchronize_session=False)
    
    # 2. 清理临时文件
    temp_cutoff = datetime.utcnow() - timedelta(hours=24)
    temp_files = find_expired_temp_files(temp_cutoff)
    for file_path in temp_files:
        try:
            os.remove(file_path)
        except OSError:
            logger.warning(f'无法删除临时文件: {file_path}')
    
    # 3. 清理数据库缓存
    cache_cutoff = datetime.utcnow() - timedelta(hours=1)
    db.session.query(StatisticsCache)\
        .filter(StatisticsCache.expires_at < cache_cutoff)\
        .delete(synchronize_session=False)
    
    db.session.commit()
```
