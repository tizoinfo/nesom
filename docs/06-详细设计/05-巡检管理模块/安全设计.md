# 巡检管理模块 - 安全详细设计

**版本**: 1.0  
**日期**: 2026-04-01  
**作者**: 智能体编排者  
**状态**: Draft  
**审核状态**: 待评审

**适用范围**: 巡检管理模块所有组件（Web前端、移动端、后端API、数据库、文件存储）

## 1. 安全设计概述

### 1.1 安全目标
巡检管理模块作为NESOM系统的预防性维护核心，处理现场巡检数据、设备状态、人员位置等敏感信息，必须满足以下安全目标：

1. **身份可信**: 确保巡检人员身份真实可靠，防止冒名顶替
2. **操作可信**: 确保巡检操作真实发生，防止虚假巡检
3. **数据可信**: 确保巡检数据真实完整，防止篡改伪造
4. **位置可信**: 确保巡检位置真实准确，防止位置欺骗
5. **隐私保护**: 保护巡检人员和设备位置隐私
6. **离线安全**: 确保离线操作数据安全，防止数据泄露

### 1.2 安全威胁模型

#### 1.2.1 威胁识别
| 威胁类型 | 威胁描述 | 影响等级 | 可能性 |
|----------|----------|----------|--------|
| 虚假巡检 | 巡检人员未到现场伪造巡检数据 | 严重 | 中 |
| 数据篡改 | 恶意修改巡检结果，掩盖问题 | 严重 | 中 |
| 位置欺骗 | 使用虚假GPS位置进行位置欺骗 | 高 | 中 |
| 离线数据泄露 | 移动端离线数据被非法获取 | 高 | 中 |
| 权限滥用 | 越权查看或修改巡检数据 | 高 | 中 |
| 中间人攻击 | 拦截篡改移动端与服务器通信 | 中 | 低 |
| 设备丢失 | 移动设备丢失导致数据泄露 | 中 | 中 |
| 拒绝服务 | 攻击导致巡检系统不可用 | 中 | 低 |

#### 1.2.2 攻击面分析
1. **移动端应用**: 离线数据存储、本地认证、文件缓存
2. **现场操作**: GPS定位、扫码验证、拍照取证
3. **数据传输**: 移动网络传输、WiFi传输、离线同步
4. **文件上传**: 现场照片、视频、录音文件
5. **API接口**: 巡检结果提交、任务状态更新
6. **Web界面**: 计划管理、任务分配、报告查看
7. **数据库**: 巡检数据存储、人员轨迹存储
8. **第三方服务**: 地图服务、短信验证、推送服务

### 1.3 安全设计原则
1. **零信任原则**: 不信任任何内部或外部实体，持续验证
2. **最小权限原则**: 用户和系统组件仅拥有必要权限
3. **纵深防御**: 多层安全防护，单一防线失效不影响整体
4. **默认安全**: 默认配置即为安全配置
5. **隐私设计**: 隐私保护融入系统设计
6. **可审计性**: 所有操作可追溯、可审计
7. **移动安全**: 充分考虑移动端特殊安全需求

## 2. 身份认证与访问控制

### 2.1 多因素身份认证

#### 2.1.1 认证因素
**知识因素 (Something you know)**:
- 用户名/密码
- PIN码
- 安全问题

** possession因素 (Something you have)**:
- 移动设备 (设备绑定)
- 数字证书
- 硬件令牌 (可选)

**生物因素 (Something you are)**:
- 指纹识别 (移动端)
- 人脸识别 (移动端)
- 声纹识别 (可选)

#### 2.1.2 认证场景设计
| 场景 | 认证要求 | 认证方式 |
|------|----------|----------|
| Web端登录 | 基础认证 | 用户名/密码 + 短信验证码 |
| 移动端登录 | 强认证 | 用户名/密码 + 设备绑定 + 生物识别 |
| 离线登录 | 本地认证 | 生物识别 + 本地PIN码 |
| 敏感操作 | 增强认证 | 操作前再次身份验证 |
| 设备更换 | 重新绑定 | 短信验证 + 管理员审核 |

#### 2.1.3 设备绑定机制
**绑定流程**:
1. 用户首次在移动端登录
2. 系统记录设备唯一标识 (IMEI/设备指纹)
3. 发送短信验证码到注册手机
4. 用户输入验证码完成绑定
5. 绑定信息加密存储到服务器

**绑定管理**:
- 单个用户最多绑定3台设备
- 绑定设备列表可查看和管理
- 设备丢失可远程解绑
- 解绑需要二次验证

### 2.2 基于角色的访问控制 (RBAC)

#### 2.2.1 角色定义
| 角色 | 描述 | 权限范围 |
|------|------|----------|
| 巡检员 | 执行现场巡检任务 | 查看分配的任务，提交巡检结果 |
| 班组长 | 管理巡检小组 | 分配任务，审核结果，查看小组报告 |
| 巡检主管 | 管理场站巡检 | 创建计划，分配任务，查看所有报告 |
| 设备管理员 | 管理设备信息 | 维护设备数据，查看设备巡检历史 |
| 系统管理员 | 系统管理 | 用户管理，角色管理，系统配置 |

#### 2.2.2 权限矩阵
| 资源/操作 | 巡检员 | 班组长 | 巡检主管 | 设备管理员 | 系统管理员 |
|-----------|--------|--------|----------|------------|------------|
| 巡检计划查看 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 巡检计划创建 | ✗ | ✗ | ✓ | ✗ | ✓ |
| 巡检计划修改 | ✗ | ✗ | ✓ | ✗ | ✓ |
| 巡检任务查看 | 自己的 | 小组的 | 全部的 | 设备相关的 | 全部的 |
| 巡检任务分配 | ✗ | 小组内 | ✓ | ✗ | ✓ |
| 巡检结果提交 | ✓ | ✓ | ✗ | ✗ | ✗ |
| 巡检结果审核 | ✗ | ✓ | ✓ | ✗ | ✓ |
| 巡检报告查看 | 自己的 | 小组的 | 全部的 | 设备相关的 | 全部的 |
| 设备信息查看 | 任务相关的 | ✓ | ✓ | ✓ | ✓ |
| 设备信息修改 | ✗ | ✗ | ✗ | ✓ | ✓ |

#### 2.2.3 数据权限控制
**基于场站的权限**:
- 用户只能访问所属场站的巡检数据
- 多场站用户需明确指定当前操作场站
- 跨场站数据访问需要特殊授权

**基于区域的权限**:
- 支持按物理区域划分数据权限
- 巡检员只能访问授权区域的设备
- 区域权限继承场站权限

**时间维度的权限**:
- 支持按时间范围控制数据访问
- 历史数据访问需要更高权限
- 实时数据与历史数据权限分离

### 2.3 会话管理

#### 2.3.1 Web端会话
- **会话令牌**: JWT (JSON Web Token)
- **令牌有效期**: 访问令牌2小时，刷新令牌7天
- **令牌刷新**: 使用刷新令牌获取新访问令牌
- **令牌撤销**: 支持主动撤销令牌
- **并发控制**: 同一用户最多3个并发会话

#### 2.3.2 移动端会话
- **长期会话**: 支持30天长期会话 (需设备绑定)
- **离线令牌**: 支持离线操作令牌
- **令牌存储**: 安全存储区 (Android KeyStore / iOS Keychain)
- **自动续期**: 网络恢复时自动续期令牌

#### 2.3.3 会话安全
- **HTTPS强制**: 所有会话必须使用HTTPS
- **令牌绑定**: 令牌与设备指纹绑定
- **防止重放**: 令牌使用时间戳和nonce
- **安全传输**: 使用HTTP Only, Secure Cookie

## 3. 数据安全

### 3.1 数据传输安全

#### 3.1.1 加密传输
**TLS配置**:
- 最低TLS 1.2，推荐TLS 1.3
- 强密码套件，禁用弱密码
- 使用可信CA证书
- 定期更新证书

**移动端特殊考虑**:
- 证书锁定 (Certificate Pinning)
- 双向TLS认证 (可选)
- 网络切换时的连接保持
- 弱网环境下的安全降级

#### 3.1.2 API安全
**请求签名**:
```typescript
// 请求签名算法
const generateSignature = (method, path, params, timestamp, nonce, secret) => {
  const paramString = sortParams(params);
  const rawString = `${method}\n${path}\n${paramString}\n${timestamp}\n${nonce}`;
  return hmacSHA256(rawString, secret);
};
```

**防护措施**:
- 接口限流: 防止暴力请求
- 请求验证: 验证时间戳、nonce、签名
- 参数过滤: 防止SQL注入、XSS攻击
- 内容检查: 检查上传文件类型和内容

### 3.2 数据存储安全

#### 3.2.1 数据库加密
**敏感字段加密**:
```sql
-- 使用数据库加密函数
CREATE TABLE inspection_results (
    id VARCHAR(36) PRIMARY KEY,
    problem_description TEXT,
    -- 加密存储
    encrypted_photos JSON ENCRYPTED USING 'AES-256-GCM',
    encryption_key_id VARCHAR(50)
);
```

**加密策略**:
- 字段级加密: 敏感字段单独加密
- 应用层加密: 数据在应用层加密后存储
- 密钥管理: 使用专门的密钥管理系统
- 密钥轮换: 定期轮换加密密钥

#### 3.2.2 文件存储安全
**现场照片加密**:
1. **移动端本地加密**:
   ```javascript
   // 拍照后立即加密
   const encryptedImage = await crypto.subtle.encrypt(
     { name: 'AES-GCM', iv: randomIV },
     encryptionKey,
     imageData
   );
   ```

2. **服务器端存储**:
   - 加密存储到对象存储
   - 访问需要临时令牌
   - 支持细粒度访问控制
   - 自动清理过期文件

**文件访问控制**:
- 基于角色的文件访问权限
- 临时访问令牌有效期5分钟
- 访问日志记录和审计
- 防止直接URL访问

#### 3.2.3 移动端离线数据安全
**本地数据库加密**:
- 使用SQLite with SQLCipher
- 256位AES加密整个数据库
- 加密密钥基于用户密码和设备标识派生
- 应用卸载时自动清理数据

**本地文件加密**:
- 现场照片、视频、录音本地加密存储
- 使用iOS Data Protection或Android EncryptedFile
- 应用后台时自动锁定加密数据
- 设备锁屏时自动加密内存中的数据

**安全存储区**:
- Android: 使用KeyStore存储加密密钥
- iOS: 使用Keychain存储加密密钥
- 防止root/jailbreak设备访问

### 3.3 数据完整性保护

#### 3.3.1 数字签名
**巡检数据签名**:
```typescript
interface InspectionResultSignature {
  resultId: string;
  inspectorId: string;
  timestamp: string;
  location: GeoPoint;
  hash: string; // 数据哈希
  signature: string; // 数字签名
  publicKey: string; // 签名公钥
}
```

**签名流程**:
1. 计算巡检数据的哈希值
2. 使用巡检员私钥对哈希值签名
3. 将签名和公钥与数据一起存储
4. 验证时使用公钥验证签名

#### 3.3.2 防篡改设计
**区块链存证** (可选):
- 重要巡检数据上链存证
- 使用私有区块链或联盟链
- 提供不可篡改的证据链
- 支持司法取证

**哈希链**:
- 每次巡检结果包含前次结果的哈希
- 形成不可篡改的哈希链
- 易于验证数据完整性
- 轻量级实现方案

## 4. 现场操作安全

### 4.1 巡检人员身份验证

#### 4.1.1 生物识别验证
**验证时机**:
- 应用启动时验证
- 开始巡检任务时验证
- 提交巡检结果时验证
- 执行敏感操作时验证

**验证方式**:
- 指纹识别 (最常用)
- 人脸识别 (支持戴口罩识别)
- 声纹识别 (特定场景)
- 多生物特征融合

#### 4.1.2 持续身份验证
**行为分析**:
- 分析巡检操作模式
- 检测异常操作行为
- 基于风险的动态验证

**设备状态监控**:
- 监控设备是否被破解
- 检测模拟器运行
- 防止自动化脚本

### 4.2 操作时间地点验证

#### 4.2.1 GPS位置验证
**防欺骗技术**:
1. **多源定位**: GPS + WiFi + 基站 + 蓝牙
2. **信号分析**: 分析GPS信号特征，检测模拟信号
3. **运动传感器**: 使用加速度计、陀螺仪验证移动
4. **环境传感器**: 使用气压计验证海拔变化

**验证规则**:
```typescript
interface LocationValidationRule {
  checkpointId: string;
  expectedLocation: GeoPoint;
  toleranceRadius: number; // 容差半径，默认50米
  requiredAccuracy: number; // 要求精度，默认10米
  minSatellites: number; // 最少卫星数，默认4
  timestampValidation: boolean; // 时间戳验证
}
```

#### 4.2.2 时间戳验证
**时间同步**:
- 移动端定期与服务器时间同步
- 使用NTP协议同步时间
- 防止设备时间被修改

**时间窗口验证**:
- 检查操作时间是否在合理范围内
- 防止重放攻击
- 支持离线时间验证

### 4.3 防篡改设计

#### 4.3.1 数据签名链
**签名链结构**:
```
任务开始签名 → 检查点1签名 → 检查点2签名 → ... → 任务完成签名
```

**签名内容**:
- 数据内容哈希
- 时间戳
- 位置信息
- 操作人员标识
- 前一个签名哈希

#### 4.3.2 证据链构建
**现场证据**:
1. **时间证据**: 可信时间戳
2. **位置证据**: 可信位置证明
3. **人员证据**: 生物识别记录
4. **设备证据**: 设备状态记录
5. **环境证据**: 现场照片、视频

**证据关联**:
- 所有证据相互关联
- 形成完整的证据链
- 支持司法取证要求

## 5. 隐私保护

### 5.1 巡检轨迹隐私保护

#### 5.1.1 轨迹数据脱敏
**脱敏策略**:
- 实时位置只保留必要精度
- 历史轨迹聚合处理
- 敏感区域轨迹模糊处理
- 支持隐私区域设置

**访问控制**:
- 轨迹数据分级访问权限
- 实时位置严格访问控制
- 历史轨迹匿名化处理
- 支持用户隐私设置

#### 5.1.2 位置隐私保护
**差分隐私**:
- 在轨迹数据中添加噪声
- 保护个人位置隐私
- 保持数据统计特性

**地理围栏隐私**:
- 隐私区域自动模糊处理
- 支持自定义隐私区域
- 合规的位置数据使用

### 5.2 个人信息保护

#### 5.2.1 数据最小化
**收集原则**:
- 只收集必要的个人信息
- 明确告知收集目的
- 获得用户明确同意
- 定期清理过期数据

**匿名化处理**:
- 统计分析使用匿名数据
- 报告中使用匿名标识
- 支持数据脱敏导出

#### 5.2.2 用户权利保障
**数据访问权**: 用户可以访问自己的数据
**数据更正权**: 用户可以更正错误数据
**数据删除权**: 用户可以要求删除数据
**数据携带权**: 用户可以导出自己的数据

**客户端执行**:
- 定期检查擦除命令
- 安全擦除数据
- 擦除结果上报
- 擦除后状态重置

## 7. 安全监控和审计

### 7.1 安全事件监控

#### 7.1.1 监控指标
**身份认证监控**:
- 失败登录次数
- 异常登录地点
- 异常登录时间
- 账户锁定事件

**操作行为监控**:
- 异常操作频率
- 权限提升尝试
- 数据访问模式
- 时间地点异常

**系统安全监控**:
- API访问频率
- 异常请求模式
- 系统资源使用
- 安全配置变更

#### 7.1.2 实时告警
**告警规则**:
```yaml
alert_rules:
  - name: "多次登录失败"
    condition: "login_failures > 5 within 5min"
    severity: "warning"
    action: ["lock_account", "notify_admin"]
    
  - name: "异常位置登录"
```

## 8. 应急响应和恢复

### 8.1 安全事件响应

#### 8.1.1 响应流程
1. **事件检测**: 监控系统发现安全事件
2. **事件确认**: 安全团队确认事件真实性
3. **事件分类**: 根据严重程度分类
4. **应急响应**: 执行应急预案
5. **事件处理**: 消除安全威胁
6. **恢复运行**: 恢复系统正常运行
7. **事后分析**: 分析原因，改进安全

#### 8.1.2 应急预案
**数据泄露预案**:
- 立即隔离受影响系统
- 评估泄露范围和影响
- 通知相关用户和监管机构
- 加强监控，防止进一步泄露

**系统入侵预案**:
- 立即切断入侵路径
- 备份系统状态证据
- 清理后门和恶意代码
- 修复安全漏洞

### 8.2 数据备份和恢复

#### 8.2.1 备份策略
**备份频率**:
- 实时备份: 关键数据实时同步
- 每日备份: 全量数据每日备份
- 每周备份: 完整系统每周备份

**备份存储**:
- 本地备份: 快速恢复
- 异地备份: 灾难恢复
- 离线备份: 防勒索软件
- 加密备份: 防止数据泄露

#### 8.2.2 恢复测试
- 每月进行恢复测试
- 验证备份数据完整性
- 测试恢复流程有效性
- 更新恢复文档
