## 1. 设计概述

### 1.1 业务领域
巡检管理模块是NESOM系统的预防性维护核心，负责新能源设备的定期检查、问题发现和维护保障。主要业务领域包括：

1. **巡检计划管理**：制定周期性、临时性巡检计划，定义巡检路线和标准
2. **任务分配调度**：基于人员、技能、位置智能分配巡检任务
3. **现场巡检执行**：移动端支持现场检查、数据采集、问题记录
4. **结果处理分析**：巡检结果分析、问题自动转工单、报告生成
5. **绩效评估优化**：巡检质量评估、人员绩效、路线优化

### 1.2 核心业务流程
1. 计划制定 → 路线规划 → 任务生成 → 智能分配 → 现场执行 → 结果提交 → 问题处理 → 报告生成
2. 模板定义 → 检查项配置 → 标准制定 → 计划关联 → 现场检查 → 结果比对 → 质量评估
3. 离线缓存 → 现场采集 → 冲突检测 → 数据同步 → 状态更新 → 实时看板

### 1.3 设计原则
1. **移动端优先**：充分考虑现场巡检的移动端需求，支持离线操作
2. **智能化**：智能路线规划、问题自动识别、人员智能匹配
3. **实时性**：任务状态实时更新，问题即时通知
4. **可靠性**：数据不丢失，支持断点续传，冲突智能解决
5. **集成性**：与设备监控、工单管理紧密集成，形成闭环

## 2. 核心业务实体状态机

### 2.1 巡检计划状态机 (Inspection Plan State Machine)

#### 2.1.1 状态定义
| 状态 | 编码 | 描述 | 业务含义 |
|------|------|------|----------|
| 草稿 | draft | 计划已创建但未启用 | 可编辑，不可生成任务 |
| 已激活 | active | 计划已启用并正在执行 | 按计划生成巡检任务 |
| 已暂停 | paused | 计划临时暂停 | 暂停生成新任务，已有任务继续执行 |
| 已完成 | completed | 计划已执行完成 | 所有任务完成，计划结束 |
| 已取消 | cancelled | 计划被取消 | 停止生成任务，取消未执行任务 |

#### 2.1.2 状态转换规则
```
      ┌─────────────┐
      │    草稿     │◀─────────────────────────────────┐
      └─────────────┘                                  │
           │ 提交审核                                    │
           ▼                                          │
      ┌─────────────┐     ┌─────────────┐           │
      │  已激活     │────▶│  已暂停     │           │
      └─────────────┘     └─────────────┘           │
           │                      │                  │
           │ 计划到期/手动完成      │ 恢复            │
           ▼                      ▼                  │
      ┌─────────────┐     ┌─────────────┐           │
      │  已完成     │     │  已激活     │           │
      └─────────────┘     └─────────────┘           │
           │                                          │
           └──────────────────────────────────────────┘
                         手动取消/系统取消
```

#### 2.1.3 状态转换条件
| 转换 | 触发条件 | 前置条件 | 后置动作 |
|------|----------|----------|----------|
| draft → active | 管理员激活计划 | 计划信息完整，路线有效 | 生成首次巡检任务，发送通知 |
| active → paused | 管理员暂停计划 | 计划处于active状态 | 停止生成新任务，通知相关人员 |
| paused → active | 管理员恢复计划 | 计划处于paused状态 | 恢复生成新任务，重新计算排期 |
| active → completed | 计划到期或手动完成 | 所有任务已完成或取消 | 更新计划完成时间，生成总结报告 |
| * → cancelled | 管理员取消计划 | 计划未完成 | 取消所有未执行任务，记录取消原因 |

### 2.2 巡检任务状态机 (Inspection Task State Machine)

#### 2.2.1 状态定义
| 状态 | 编码 | 描述 | 业务含义 |
|------|------|------|----------|
| 待分配 | pending | 任务已创建，等待分配 | 系统自动分配或管理员手动分配 |
| 已分配 | assigned | 任务已分配给人 | 等待巡检人员开始执行 |
| 进行中 | in_progress | 巡检人员已开始执行 | 正在现场巡检 |
| 已完成 | completed | 巡检任务完成 | 所有检查点完成，结果已提交 |
| 已取消 | cancelled | 任务被取消 | 任务不再执行 |
| 已过期 | overdue | 任务超过计划时间未开始 | 需要重新安排或升级处理 |

#### 2.2.2 状态转换规则
```
      ┌─────────────┐
      │  待分配     │
      └─────────────┘
           │ 自动分配/手动分配
           ▼
      ┌─────────────┐     ┌─────────────┐
      │  已分配     │────▶│  已过期     │
      └─────────────┘     └─────────────┘
           │                      │
           │ 开始巡检              │ 重新分配
           ▼                      ▼
      ┌─────────────┐     ┌─────────────┐
      │  进行中     │     │  已分配     │
      └─────────────┘     └─────────────┘
           │
           │ 提交结果
           ▼
      ┌─────────────┐
      │  已完成     │
      └─────────────┘
           │
           │ 取消任务
           ▼
      ┌─────────────┐
      │  已取消     │
      └─────────────┘
```

#### 2.2.3 状态转换条件
| 转换 | 触发条件 | 前置条件 | 后置动作 |
|------|----------|----------|----------|
| pending → assigned | 系统自动分配或管理员手动分配 | 任务处于pending状态 | 记录分配时间和人员，发送通知 |
| assigned → in_progress | 巡检人员开始任务 | 任务处于assigned状态 | 记录开始时间、位置，更新任务状态 |
| in_progress → completed | 巡检人员提交所有结果 | 所有检查点完成 | 计算完成率、耗时，生成初步报告 |
| assigned → overdue | 任务超过计划时间未开始 | 当前时间 > scheduled_date + 1天 | 发送逾期通知，升级处理 |
| overdue → assigned | 重新分配任务 | 任务处于overdue状态 | 重新分配给人，更新计划时间 |
| * → cancelled | 管理员取消任务 | 任务未完成 | 记录取消原因，释放人员工作负荷 |

## 3. 核心业务逻辑设计

### 3.1 巡检计划生成逻辑

#### 3.1.1 基于设备类型和位置的自动路线生成
**业务目标**: 根据设备分布自动生成最优巡检路线

**算法流程**:
1. **设备聚类分析**: 根据设备位置进行空间聚类，形成巡检区域
2. **路线优化算法**: 使用旅行商问题(TSP)算法或遗传算法优化路线
3. **检查点排序**: 考虑设备重要性、检查耗时、安全要求等因素排序
4. **路径计算**: 集成地图服务计算实际行走路径和距离
5. **时间预估**: 根据检查点数量和预估耗时计算总巡检时间

**关键参数**:
- 设备位置坐标 (latitude, longitude)
- 设备重要性权重 (1-10)
- 单点检查预估时间 (5-30分钟)
- 移动平均速度 (步行3km/h, 车辆30km/h)
- 最大连续工作时间 (4小时)

**输出结果**:
- 巡检路线定义 (inspection_routes)
- 检查点序列 (inspection_checkpoints)
- 预估总距离和耗时

#### 3.1.2 巡检频率和优先级计算
**频率计算规则**:
1. **基于设备类型**: 关键设备(如逆变器)每周，普通设备(如光伏板)每月
2. **基于运行状态**: 故障率高的设备增加巡检频率
3. **基于季节因素**: 夏季增加温度相关检查频率
4. **基于历史问题**: 曾出问题的设备增加关注频率

**优先级确定规则**:
1. **设备重要性**: 关键设备优先级高
2. **问题严重性**: 近期有告警的设备优先级高
3. **时间敏感性**: 临近保养期的设备优先级高
4. **季节性因素**: 季节相关检查按时令安排

#### 3.1.3 周期性计划和临时计划支持
**周期性计划**:
- 支持日、周、月、季度、年等周期
- 支持自定义周期 (如每10天)
- 支持工作日/周末区分
- 支持节假日特殊安排

**临时计划**:
- 应急检查: 设备故障后专项检查
- 专项检查: 特定问题排查
- 陪同检查: 领导或客户参观检查
- 演练检查: 应急演练检查

### 3.2 巡检任务分配逻辑

#### 3.2.1 基于人员技能和位置的智能分配
**分配策略**:
1. **轮询分配 (Round Robin)**: 平均分配任务，保证工作量均衡
2. **技能匹配 (Skill Based)**: 根据人员技能分配专业任务
3. **位置优化 (Location Based)**: 就近分配，减少移动时间
4. **负荷均衡 (Load Balancing)**: 考虑人员当前任务量
5. **手动分配 (Manual)**: 管理员手动指定

**技能匹配算法**:
```
For each task:
  1. 确定任务所需技能 (如: 光伏设备检查、电气安全)
  2. 筛选具备所需技能的人员
  3. 计算人员当前位置到任务起点的距离
  4. 考虑人员当前任务负荷
  5. 计算综合得分: 技能匹配度 × 0.4 + 位置得分 × 0.3 + 负荷得分 × 0.3
  6. 选择得分最高的人员
```

**位置优化算法**:
- 使用Haversine公式计算两点间距离
- 考虑交通方式 (步行、车辆)
- 考虑场站内路径可达性
- 批量任务分配时使用VRP(车辆路径问题)算法

#### 3.2.2 任务量和时间窗口考虑
**工作量计算**:
- 每日最大任务数: 根据人员能力设定 (默认10个任务/天)
- 任务耗时预估: 检查点数量 × 平均检查时间 + 移动时间
- 工作时段: 考虑人员排班和工作时间

**时间窗口约束**:
- 任务必须在计划时间窗口内执行
- 支持弹性时间窗口 (如: 08:00-12:00)
- 支持连续任务时间安排
- 考虑任务间的移动时间

#### 3.2.3 手动调整和重新分配
**手动调整场景**:
1. 人员请假或调岗
2. 任务优先级变化
3. 突发情况需要重新分配
4. 人员技能不足需要更换

**重新分配流程**:
1. 释放原分配任务
2. 重新计算可分配人员
3. 发送重新分配通知
4. 记录分配变更历史

### 3.3 现场巡检流程

#### 3.3.1 扫码确认设备
**扫码方式**:
- 二维码扫码: 设备二维码包含设备ID、类型、位置信息
- NFC感应: 支持NFC标签感应确认
- 手动选择: 从设备列表中选择

**确认逻辑**:
1. 扫码获取设备ID
2. 验证设备是否在巡检任务检查点列表中
3. 验证设备位置与检查点位置匹配 (GPS验证)
4. 记录到达时间、位置
5. 显示设备信息和检查项

**防作弊机制**:
- GPS位置验证: 当前位置与设备位置距离 < 容差半径(默认50米)
- 时间合理性验证: 检查时间在合理范围内
- 顺序验证: 检查点顺序执行验证 (可配置是否强制)

#### 3.3.2 逐项检查并记录
**检查项类型**:
1. **视觉检查**: 外观、清洁度、损坏情况
2. **仪表读数**: 温度、压力、电压、电流等
3. **功能测试**: 开关、按钮、指示灯测试
4. **安全检查**: 接地、绝缘、防护装置
5. **环境检查**: 周围环境、通风、排水

**记录方式**:
- 单选/多选: 正常/异常/不适用
- 数值输入: 输入具体测量值
- 文本描述: 问题详细描述
- 照片证据: 现场拍照记录
- 语音备注: 录音说明

**数据验证**:
- 范围验证: 数值在合理范围内
- 必填验证: 必填项必须填写
- 逻辑验证: 前后数据逻辑一致性

#### 3.3.3 拍照/录音/视频记录
**多媒体记录规范**:
1. **照片要求**:
   - 格式: JPEG/PNG
   - 大小: 每张不超过5MB
   - 数量: 每个检查点最多10张
   - 内容: 包含设备全景、特写、问题部位
   - 元数据: 包含GPS位置、时间戳、方向

2. **录音要求**:
   - 格式: MP3/AAC
   - 时长: 每段不超过3分钟
   - 内容: 问题描述、现场情况说明

3. **视频要求**:
   - 格式: MP4
   - 时长: 每段不超过2分钟
   - 内容: 动态问题记录、操作过程

**存储策略**:
- 本地缓存: 原始文件本地存储
- 压缩上传: 上传时自动压缩
- 分级存储: 热数据SSD，冷数据HDD
- 备份策略: 多地备份，定期归档

#### 3.3.4 GPS位置和时间戳
**GPS数据采集**:
- 采集频率: 每30秒采集一次位置
- 精度要求: 水平精度 < 10米
- 数据格式: WGS84坐标系 (经度, 纬度, 海拔)
- 补充数据: 速度、方向、精度半径

**时间戳规范**:
- 时间标准: UTC时间，时区转换在应用层处理
- 精度: 毫秒级精度
- 同步机制: 定期与服务器时间同步
- 防篡改: 使用数字签名保护时间戳

**轨迹记录**:
- 轨迹点: 按时间顺序记录位置点
- 轨迹分析: 计算移动距离、速度、停留时间
- 轨迹可视化: 在地图上显示巡检路径
- 异常检测: 检测异常停留、偏离路线

### 3.4 巡检结果处理逻辑

#### 3.4.1 自动问题识别（与阈值比较）
**阈值比对算法**:
```python
def check_threshold(actual_value, threshold_config):
    """
    检查实际值是否超过阈值
    """
    if threshold_config['type'] == 'min':
        if actual_value < threshold_config['warning']:
            return 'warning', f"值{actual_value}低于警告阈值{threshold_config['warning']}"
        elif actual_value < threshold_config['error']:
            return 'error', f"值{actual_value}低于错误阈值{threshold_config['error']}"
    
    elif threshold_config['type'] == 'max':
        if actual_value > threshold_config['warning']:
            return 'warning', f"值{actual_value}高于警告阈值{threshold_config['warning']}"
        elif actual_value > threshold_config['error']:
            return 'error', f"值{actual_value}高于错误阈值{threshold_config['error']}"
    
    return 'normal', None
```

**多维度问题识别**:
1. **阈值超标**: 数值超过预设阈值
2. **趋势异常**: 与历史数据趋势不符
3. **关联异常**: 关联设备数据不一致
4. **规则触发**: 满足自定义规则条件
5. **AI识别**: 基于图像识别的异常检测

**问题分级**:
- **轻微问题**: 不影响运行，需关注
- **一般问题**: 影响效率，需计划处理
- **严重问题**: 影响安全，需立即处理
- **紧急问题**: 可能导致事故，需紧急处理

#### 3.4.2 问题自动转工单
**转工单规则**:
1. **自动转单条件**:
   - 问题级别为严重或紧急
   - 相同问题重复出现
   - 设备重要程度高
   - 用户配置的自动转单规则

2. **工单信息生成**:
   - 标题: 基于问题描述自动生成
   - 内容: 包含巡检结果、设备信息、现场照片
   - 优先级: 根据问题级别确定
   - 分类: 根据问题类型确定
   - 分配: 根据维修人员技能和位置分配

3. **关联关系建立**:
   - 工单与巡检任务关联
   - 工单与设备关联
   - 工单与检查点关联
   - 支持双向追溯

**工单状态同步**:
- 工单创建时通知相关人员
- 工单状态更新同步到巡检记录
- 工单处理结果反馈到设备健康状态
- 工单完成关闭关联巡检问题

#### 3.4.3 巡检报告自动生成
**报告类型**:
1. **任务报告**: 单次巡检任务详细报告
2. **日报/周报/月报**: 周期性巡检总结报告
3. **专项报告**: 特定设备或问题专项报告
4. **绩效报告**: 巡检人员绩效报告
5. **分析报告**: 巡检数据分析报告

**报告内容**:
- 执行概况: 任务完成情况、耗时、问题统计
- 问题分析: 问题分布、趋势、根本原因
- 设备健康: 设备健康评分、变化趋势
- 人员绩效: 完成率、效率、质量问题
- 改进建议: 基于数据分析的改进建议

**生成时机**:
- 任务完成时自动生成任务报告
- 每天凌晨生成前一日日报
- 每周一生成上周周报
- 每月1日生成上月月报
- 按需手动生成专项报告

### 3.5 移动端离线支持

#### 3.5.1 离线数据缓存
**缓存策略**:
1. **预缓存数据**:
   - 用户相关: 个人信息、权限
   - 任务数据: 已分配的未来7天任务
   - 模板数据: 常用巡检模板
   - 设备数据: 任务相关设备信息
   - 地图数据: 离线地图切片

2. **缓存更新机制**:
   - 增量更新: 只同步变更数据
   - 定时更新: 每天凌晨自动更新
   - 手动更新: 用户手动触发更新
   - 智能更新: 根据网络状况智能更新

3. **缓存管理**:
   - 容量控制: 最大缓存容量1GB
   - 清理策略: LRU(最近最少使用)清理
   - 版本管理: 数据版本一致性检查
   - 加密存储: 本地缓存数据加密

**离线操作支持**:
- 离线查看: 查看任务、设备、模板信息
- 离线执行: 执行巡检任务，记录结果
- 离线拍照: 拍照并本地存储
- 离线记录: 记录检查结果、问题描述

#### 3.5.2 冲突检测和合并
**冲突类型**:
1. **数据修改冲突**: 同一数据在本地和服务器都被修改
2. **删除冲突**: 本地删除但服务器修改，或反之
3. **关联冲突**: 关联数据不一致
4. **业务规则冲突**: 违反业务规则

**冲突检测算法**:
```
冲突检测流程:
  1. 为每条记录维护版本号(version)和最后修改时间(last_modified)
  2. 同步时比较本地和服务器版本
  3. 如果版本不同且最后修改时间不同，检测到冲突
  4. 根据冲突类型应用不同解决策略
```

**冲突解决策略**:
1. **服务器优先**: 服务器数据覆盖本地数据
2. **客户端优先**: 本地数据覆盖服务器数据
3. **人工解决**: 需要人工介入解决
4. **合并策略**: 智能合并可合并字段

**解决流程**:
```
冲突解决流程:
  1. 检测到冲突，暂停同步
  2. 显示冲突详情给用户
  3. 用户选择解决策略或人工解决
  4. 应用解决策略，继续同步
  5. 记录冲突解决日志
```

#### 3.5.3 断点续传
**文件断点续传**:
1. **分片上传**: 大文件分片上传，每片2MB
2. **断点记录**: 记录已成功上传的分片
3. **续传机制**: 续传时从断点处继续
4. **完整性校验**: 上传完成后校验文件完整性

**数据同步断点续传**:
1. **同步状态记录**: 记录同步进度和状态
2. **事务性同步**: 确保数据同步的原子性
3. **错误重试**: 网络错误时自动重试，最大重试3次
4. **进度保存**: 同步进度本地保存，应用重启后可继续

**网络状态检测**:
- 实时检测网络连接状态
- 预测网络带宽，优化同步策略
- 网络恢复时自动触发同步
- 弱网环境下使用压缩和增量同步

## 5. 异常处理机制

### 5.1 业务异常
**异常类型**:
1. **数据异常**: 数据不一致、格式错误
2. **流程异常**: 流程状态错误、顺序错误
3. **权限异常**: 无权限操作、越权访问
4. **业务逻辑异常**: 违反业务规则

**处理策略**:
- 数据异常: 数据校验、清理、修复
- 流程异常: 状态修复、流程回退
- 权限异常: 权限检查、访问控制
- 业务逻辑异常: 规则检查、人工干预

### 5.2 系统异常
**异常类型**:
1. **网络异常**: 连接超时、网络中断
2. **数据库异常**: 连接失败、死锁、超时
3. **外部服务异常**: 地图服务、短信服务异常
4. **资源异常**: 内存不足、磁盘满

**处理策略**:
- 网络异常: 重试机制、离线模式
- 数据库异常: 连接池管理、事务回滚
- 外部服务异常: 降级处理、备用服务
- 资源异常: 资源监控、自动告警

### 5.3 巡检过程异常
**异常场景**:
1. **设备无法定位**: GPS信号弱，无法验证位置
2. **检查项不适用**: 设备状态变化导致检查项不适用
3. **数据异常值**: 测量值明显异常
4. **流程中断**: 巡检过程被意外中断

**处理流程**:
1. 记录异常情况，包括时间、位置、原因
2. 根据异常类型应用处理策略
3. 需要时通知相关人员
4. 生成异常报告，用于后续分析
