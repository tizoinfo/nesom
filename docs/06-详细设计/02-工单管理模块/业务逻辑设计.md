## 1. 设计概述

### 1.1 业务领域
工单管理模块是NESOM系统的业务核心，负责新能源设备维护工单的全生命周期管理。主要业务领域包括：

1. **工单创建**：基于设备告警、巡检计划或手动创建工单
2. **工单分配**：智能调度和人工分配执行人员
3. **工单处理**：现场作业记录、备件消耗、质量检查
4. **工单审核**：处理结果审核、成本核算、质量评估
5. **工单关闭**：归档、评价、知识积累
6. **工单分析**：绩效统计、效率分析、成本分析

### 1.2 核心业务流程
1. 故障发生 → 告警生成 → 工单创建 → 工单分配 → 现场处理 → 审核关闭
2. 巡检计划 → 计划工单 → 分配执行 → 现场巡检 → 结果录入 → 报告生成
3. 预防维护 → 维护工单 → 分配执行 → 维护作业 → 设备测试 → 记录归档
4. 工单完成 → 客户评价 → 绩效计算 → 知识积累 → 流程优化

### 1.3 设计原则
1. **流程完整性**：覆盖工单从创建到归档的全流程
2. **数据一致性**：事务保障工单与关联数据的一致性
3. **移动端优先**：支持移动端扫码、拍照、离线操作
4. **智能化调度**：基于人员技能、位置、负载智能分配
5. **安全合规**：操作可追溯，数据可审计，权限精细化

## 2. 核心业务实体状态机

### 2.1 工单状态机 (Work Order Status State Machine)

#### 2.1.1 状态定义
| 状态 | 编码 | 描述 | 业务含义 |
|------|------|------|----------|
| 草稿 | draft | 工单创建中，未提交 | 可编辑，可删除，未进入流程 |
| 待分配 | pending | 已提交，等待分配执行人 | 不可编辑，等待调度员分配 |
| 已分配 | assigned | 已分配执行人，等待开始 | 执行人确认后可开始处理 |
| 进行中 | in_progress | 执行人已开始处理 | 现场作业中，可记录处理步骤 |
| 待审核 | pending_review | 处理完成，等待审核 | 审核人检查处理结果和质量 |
| 已完成 | completed | 审核通过，工单完成 | 可进行评价，准备关闭 |
| 已关闭 | closed | 工单关闭，进入历史 | 只读状态，不可修改 |
| 已取消 | cancelled | 工单取消 | 终止流程，记录取消原因 |

#### 2.1.2 状态转换规则
```
                                     ┌─────────────┐
                                     │   draft     │◄───创建工单
                                     └──────┬──────┘
                                            │ 提交
                                            ▼
                                     ┌─────────────┐
                                     │  pending    │◄───告警转工单
                                     └──────┬──────┘
                                            │ 分配
                                            ▼
                                     ┌─────────────┐
                                     │  assigned   │
                                     └──────┬──────┘
                                            │ 开始处理
                                            ▼
                                     ┌─────────────┐
                                     │ in_progress │
                                     └──────┬──────┘
                                            │ 提交审核
                                            ▼
                                     ┌─────────────┐
        取消─────┐               │pending_review│
                 │                    └──────┬──────┘
                 │                           │ 审核通过
                 │                           ▼
                 │                    ┌─────────────┐
                 └─────────────────── │ completed   │
                                      └──────┬──────┘
                                             │ 关闭
                                             ▼
                                      ┌─────────────┐
                                      │   closed    │
                                      └─────────────┘
```

#### 2.1.3 状态转换约束
| 转换 | 原状态 | 新状态 | 触发条件 | 必填信息 | 权限要求 |
|------|--------|--------|----------|----------|----------|
| 提交 | draft | pending | 工单信息完整 | - | 创建者/编辑者 |
| 分配 | pending | assigned | 选择执行人 | 执行人、计划时间 | 调度员/管理员 |
| 开始 | assigned | in_progress | 执行人确认 | 开始时间、位置 | 分配的执行人 |
| 提交审核 | in_progress | pending_review | 处理完成 | 完成时间、处理结果 | 执行人 |
| 退回修改 | pending_review | in_progress | 审核不通过 | 审核意见、修改要求 | 审核人 |
| 审核通过 | pending_review | completed | 审核通过 | 审核意见、实际成本 | 审核人 |
| 关闭 | completed | closed | 确认关闭 | 关闭原因 | 创建者/审核人/管理员 |
| 取消 | draft/pending/assigned | cancelled | 取消工单 | 取消原因 | 创建者/调度员/管理员 |

### 2.2 工单优先级管理

#### 2.2.1 优先级定义
| 优先级 | 处理时限 | 升级规则 | 通知方式 |
|--------|----------|----------|----------|
| 低 (low) | 72小时 | 超时48小时提醒 | 站内信 |
| 中 (medium) | 24小时 | 超时12小时升级 | 站内信+邮件 |
| 高 (high) | 8小时 | 超时4小时升级 | 站内信+邮件+短信 |
| 紧急 (emergency) | 2小时 | 超时1小时通知主管 | 站内信+邮件+短信+电话 |

#### 2.2.2 优先级自动调整规则
1. **基于紧急程度**：
   - 紧急程度critical自动设为emergency优先级
   - 紧急程度urgent自动设为high优先级
   - 紧急程度normal根据其他因素确定

2. **基于设备影响**：
   - 关键设备（如主变压器）故障自动提高优先级
   - 影响发电量的设备故障提高优先级
   - 安全相关设备故障设为最高优先级

3. **基于时间因素**：
   - 夜间、节假日发生的故障提高优先级
   - 已超时工单自动升级优先级

## 3. 业务规则引擎

### 3.1 工单分配规则

#### 3.1.1 分配策略
```python
class WorkOrderAssignmentEngine:
    def assign_work_order(self, work_order, available_technicians):
        """智能分配工单给最合适的技工"""
        
        scores = {}
        for tech in available_technicians:
            score = 0
            
            # 1. 技能匹配度 (40%)
            skill_match = self.calculate_skill_match(work_order, tech)
            score += skill_match * 0.4
            
            # 2. 地理位置接近度 (30%)
            location_proximity = self.calculate_location_proximity(work_order, tech)
            score += location_proximity * 0.3
            
            # 3. 当前工作负载 (20%)
            workload_factor = self.calculate_workload_factor(tech)
            score += workload_factor * 0.2
            
            # 4. 历史绩效 (10%)
            performance_score = self.get_performance_score(tech)
            score += performance_score * 0.1
            
            scores[tech.id] = score
        
        # 返回得分最高的技工
        return max(scores.items(), key=lambda x: x[1])[0]
```

#### 3.1.2 分配约束
1. **技能约束**：维修光伏逆变器需要光伏维修资质
2. **安全约束**：高压设备维修需要高压作业证书
3. **地域约束**：优先分配给设备所在场站的驻场人员
4. **时间约束**：考虑人员排班和休息时间

### 3.2 工单时效监控规则

#### 3.2.1 超时检测
```python
def check_work_order_timeout(work_order):
    """检查工单是否超时"""
    
    now = datetime.now()
    
    if work_order.status in ['assigned', 'in_progress']:
        # 计算已分配时间
        assigned_hours = (now - work_order.assigned_at).total_seconds() / 3600
        
        # 根据优先级判断是否超时
        timeout_hours = {
            'emergency': 2,
            'high': 8,
            'medium': 24,
            'low': 72
        }
        
        if assigned_hours > timeout_hours[work_order.priority]:
            return True, f"工单已超时{assigned_hours - timeout_hours[work_order.priority]:.1f}小时"
    
    return False, None
```

#### 3.2.2 自动升级规则
1. **优先级升级**：超时50%自动提升一个优先级等级
2. **通知升级**：超时后每2小时增加一种通知方式
3. **负责人升级**：严重超时通知主管和场站负责人

### 3.3 成本控制规则

#### 3.3.1 成本预估规则
1. **人工成本**：标准工时 × 人员等级费率
2. **备件成本**：备件单价 × 数量 × 损耗系数
3. **工具成本**：设备使用费 + 耗材费
4. **其他成本**：交通费、住宿费、外包费

#### 3.3.2 成本超支预警
```python
def check_cost_overrun(work_order):
    """检查成本是否超支"""
    
    if work_order.actual_cost and work_order.cost_estimate:
        overrun_percentage = (work_order.actual_cost - work_order.cost_estimate) / work_order.cost_estimate
        
        if overrun_percentage > 0.1:  # 超支10%
            return True, f"成本超支{overrun_percentage*100:.1f}%"
        elif overrun_percentage > 0.2:  # 超支20%
            return True, f"严重成本超支{overrun_percentage*100:.1f}%，需审批"
    
    return False, None
```

## 4. 业务异常处理

### 4.1 常见异常场景

#### 4.1.1 人员不可用
- **场景**：分配的执行人请假、离职或已满负荷
- **处理**：
  1. 自动重新分配其他可用人员
  2. 通知调度员人工干预
  3. 记录分配异常原因

#### 4.1.2 备件不足
- **场景**：维修所需备件库存不足
- **处理**：
  1. 查找替代备件或调拨方案
  2. 紧急采购流程
  3. 工单暂停等待备件

#### 4.1.3 现场情况变化
- **场景**：现场发现新问题或原方案不可行
- **处理**：
  1. 执行人提交变更申请
  2. 技术负责人审批新方案
  3. 更新工单信息和预估

#### 4.1.4 质量不达标
- **场景**：审核发现处理质量不合格
- **处理**：
  1. 退回重新处理
  2. 记录质量问题
  3. 扣减执行人绩效分

### 4.2 异常处理流程
```
异常发生
   ↓
异常分类（人员/物料/技术/质量）
   ↓
自动处理尝试
   ↓
是否需要人工干预？
   ↓
是 → 通知相关人员 → 人工处理 → 记录处理结果
   ↓
否 → 系统自动处理 → 记录处理日志
   ↓
更新工单状态
   ↓
异常关闭
```

## 5. 业务数据校验规则

### 5.1 工单创建校验
1. **必填字段**：标题、描述、设备、上报人
2. **设备状态**：设备必须存在且未报废
3. **人员权限**：上报人必须有创建工单权限
4. **时间逻辑**：计划开始时间不能在计划结束时间之后

### 5.2 工单分配校验
1. **执行人状态**：执行人必须在职且有相应技能
2. **时间冲突**：执行人在计划时间内无其他已分配工单
3. **位置合理**：执行人位置与工单位置距离可接受

### 5.3 工单处理校验
1. **步骤完整性**：关键步骤必须完成并记录
2. **备件消耗**：领用备件必须记录数量和用途
3. **质量检查**：关键质量检查点必须通过
4. **安全规范**：安全措施必须执行并记录

### 5.4 工单审核校验
1. **处理结果**：必须有明确的问题原因和处理方法
2. **测试数据**：修复后测试数据必须达标
3. **成本合理**：实际成本必须在合理范围内
4. **文档完整**：所有要求的文档必须齐全

## 6. 业务统计指标

### 6.1 效率指标
1. **平均响应时间**：从创建到分配的间隔
2. **平均处理时间**：从分配到完成的总时长
3. **按时完成率**：在规定时限内完成的工单比例
4. **首次修复率**：一次修复成功的比例

### 6.2 质量指标
1. **客户满意度**：评价平均分和好评率
2. **返工率**：需要重新处理的工单比例
3. **质量合格率**：一次审核通过的比例
4. **投诉率**：产生投诉的工单比例

### 6.3 成本指标
1. **平均单次成本**：工单平均花费
2. **成本超支率**：超预算工单比例
3. **备件使用率**：备件消耗与库存比例
4. **人工效率**：产值与人工成本比率

### 6.4 人员绩效指标
1. **个人完成量**：月度/季度完成工单数
2. **个人满意度**：负责工单的平均评价分
3. **个人效率**：平均处理时间与团队平均对比
4. **技能提升**：掌握的技能种类和等级
