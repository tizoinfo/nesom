## 1. 设计概述

### 1.1 设计目标
系统集成数据库设计旨在支持NESOM系统7个MVP模块间的数据共享、一致性和高效交互，包括：
- 定义跨模块共享的核心实体和关系
- 设计模块间数据交换的标准化格式和存储
- 实现数据同步和事件驱动的集成机制
- 支持微服务架构下的分布式数据管理
- 提供系统级数据监控和审计能力

### 1.2 设计原则
1. **解耦性**：模块间通过标准化接口交互，避免直接数据库耦合
2. **一致性**：保障跨模块数据最终一致性
3. **可追溯性**：完整记录模块间数据流动和转换过程
4. **高性能**：优化集成数据查询和同步性能
5. **可扩展性**：支持新模块接入和集成模式扩展
6. **监控友好**：内置集成健康状态监控指标

### 1.3 集成架构模式
采用**事件驱动架构**为主、**API网关**为辅的混合集成模式：
- **事件总线**：模块间通过事件异步通信
- **共享内核**：核心实体在模块间共享数据模型
- **API网关**：外部系统通过统一网关访问
- **数据同步**：关键数据定期同步保障一致性

### 1.4 与其他模块关系
| 模块名称 | 集成类型 | 数据流向 | 同步机制 |
|----------|----------|----------|----------|
| 设备监控模块 | 事件生产者/消费者 | 双向 | 实时事件 + 定期快照 |
| 数据采集模块 | 事件生产者 | 单向 -> | 实时流式数据 |
| 用户权限管理 | 共享内核 | 双向 | 实时同步 |
| 报警管理模块 | 事件消费者 | 单向 <- | 实时事件订阅 |
| 巡检管理模块 | 事件生产者/消费者 | 双向 | 事件驱动 + 数据同步 |
| 报表统计模块 | 数据消费者 | 单向 <- | 批量数据同步 |
| 系统配置模块 | 共享内核 | 双向 | 实时同步 |

## 2. 集成核心表结构详细设计

### 2.1 集成事件表 (integration_events)
记录所有模块间交互的事件，支持事件溯源和重放。

| 字段名 | 数据类型 | 约束 | 默认值 | 说明 |
|--------|----------|------|--------|------|
| id | VARCHAR(36) | PRIMARY KEY | UUID() | 事件唯一标识 |
| event_id | VARCHAR(100) | UNIQUE NOT NULL | - | 业务事件ID，格式：MODULE-ENTITY-ACTION-TIMESTAMP |
| event_type | VARCHAR(50) | NOT NULL INDEX | - | 事件类型：entity_created/entity_updated/entity_deleted/state_changed/action_completed |
| source_module | VARCHAR(30) | NOT NULL INDEX | - | 事件来源模块：device_monitoring/data_collection/user_mgmt/alarm_mgmt/inspection_mgmt/report_mgmt/system_config |
| target_module | VARCHAR(30) | INDEX | NULL | 目标模块（点对点事件时） |
| entity_type | VARCHAR(50) | NOT NULL INDEX | - | 实体类型：device/device_metric/user/role/alarm/inspection_task/report_template |
| entity_id | VARCHAR(36) | NOT NULL INDEX | - | 实体ID |
| event_version | INT | NOT NULL | 1 | 事件版本，支持架构演进 |
| event_payload | JSON | NOT NULL | - | 事件完整数据负载 |
| metadata | JSON | - | NULL | 事件元数据：触发用户、IP、请求ID等 |
| published_at | DATETIME(3) | NOT NULL INDEX | CURRENT_TIMESTAMP(3) | 事件发布时间 |
| consumed_at | DATETIME(3) | - | NULL | 事件消费时间 |
| consumer_count | INT | NOT NULL | 0 | 已消费模块计数 |
| status | ENUM('pending','processing','completed','failed','archived') | NOT NULL INDEX | 'pending' | 事件状态 |
| error_message | TEXT | - | NULL | 处理失败时的错误信息 |
| retry_count | INT | NOT NULL | 0 | 重试次数 |
| created_at | DATETIME | NOT NULL | CURRENT_TIMESTAMP | 记录创建时间 |
| updated_at | DATETIME | NOT NULL | CURRENT_TIMESTAMP ON UPDATE | 最后更新时间 |

**索引设计**：
- 主键索引：`PRIMARY KEY (id)`
- 唯一索引：`UNIQUE KEY uk_event_id (event_id)`
- 查询索引：`INDEX idx_source_module (source_module, published_at DESC)`, `INDEX idx_entity (entity_type, entity_id)`, `INDEX idx_status_published (status, published_at)`
- 复合索引：`INDEX idx_module_entity (source_module, entity_type, entity_id)` 用于模块实体变更追溯

**业务规则**：
1. 事件ID生成规则：`{source_module_abbr}-{entity_type}-{action}-{timestamp_ms}-{random6}`
2. 事件至少保留30天，重要事件永久归档
3. 失败事件自动重试3次，间隔指数退避
4. 事件状态流转：pending → processing → completed/failed → archived

### 2.2 数据同步表 (data_sync_records)
记录模块间数据同步状态，保障数据最终一致性。

| 字段名 | 数据类型 | 约束 | 默认值 | 说明 |
|--------|----------|------|--------|------|
| id | BIGINT | PRIMARY KEY AUTO_INCREMENT | - | 自增主键 |
| sync_job_id | VARCHAR(36) | NOT NULL INDEX | UUID() | 同步作业ID，一次同步任务一个ID |
| source_module | VARCHAR(30) | NOT NULL INDEX | - | 数据源模块 |
| target_module | VARCHAR(30) | NOT NULL INDEX | - | 目标模块 |
| sync_type | ENUM('full','incremental','delta','real_time') | NOT NULL | 'incremental' | 同步类型 |
| entity_type | VARCHAR(50) | NOT NULL INDEX | - | 同步的实体类型 |
| filter_criteria | JSON | - | NULL | 过滤条件 |
| total_records | INT | - | NULL | 总记录数 |
| synced_records | INT | NOT NULL | 0 | 已同步记录数 |
| failed_records | INT | NOT NULL | 0 | 失败记录数 |
| sync_start_time | DATETIME(3) | NOT NULL INDEX | CURRENT_TIMESTAMP(3) | 同步开始时间 |
| sync_end_time | DATETIME(3) | - | NULL | 同步结束时间 |
| status | ENUM('pending','running','completed','failed','cancelled') | NOT NULL INDEX | 'pending' | 同步状态 |
| error_details | JSON | - | NULL | 错误详情 |
| retry_count | INT | NOT NULL | 0 | 重试次数 |
| next_retry_time | DATETIME | - | NULL | 下次重试时间 |
| created_by | VARCHAR(36) | NOT NULL | 'system' | 触发者（用户ID或system） |
| created_at | DATETIME | NOT NULL | CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | NOT NULL | CURRENT_TIMESTAMP ON UPDATE | 更新时间 |

**索引设计**：
- 主键索引：`PRIMARY KEY (id)`
- 查询索引：`INDEX idx_sync_job (sync_job_id)`, `INDEX idx_module_pair (source_module, target_module, sync_start_time DESC)`
- 状态索引：`INDEX idx_status_start (status, sync_start_time)` 用于监控长时间运行作业

**业务规则**：
1. 全量同步：每天凌晨2点执行，同步昨日变更
2. 增量同步：每5分钟执行，同步实时变更
3. 实时同步：通过事件驱动，延迟<1秒
4. 同步失败告警：连续3次失败触发告警

### 2.3 API调用日志表 (api_call_logs)
记录所有跨模块API调用，用于监控、审计和故障排查。

| 字段名 | 数据类型 | 约束 | 默认值 | 说明 |
|--------|----------|------|--------|------|
| id | BIGINT | PRIMARY KEY AUTO_INCREMENT | - | 自增主键 |
| request_id | VARCHAR(36) | UNIQUE NOT NULL INDEX | UUID() | 请求唯一标识 |
| caller_module | VARCHAR(30) | NOT NULL INDEX | - | 调用方模块 |
| callee_module | VARCHAR(30) | NOT NULL INDEX | - | 被调用模块 |
| api_endpoint | VARCHAR(255) | NOT NULL | - | API端点路径 |
| http_method | VARCHAR(10) | NOT NULL | - | HTTP方法：GET/POST/PUT/DELETE/PATCH |
| request_headers | JSON | - | NULL | 请求头信息 |
| request_body | JSON | - | NULL | 请求体（敏感信息脱敏） |
| request_size | INT | NOT NULL | 0 | 请求大小（字节） |
| response_status | INT | NOT NULL INDEX | - | HTTP响应状态码 |
| response_body | JSON | - | NULL | 响应体（截断，只存关键信息） |
| response_size | INT | NOT NULL | 0 | 响应大小（字节） |
| start_time | DATETIME(6) | NOT NULL INDEX | CURRENT_TIMESTAMP(6) | 请求开始时间 |
| end_time | DATETIME(6) | - | NULL | 响应结束时间 |
| duration_ms | INT | - | NULL | 调用耗时（毫秒） |
| client_ip | VARCHAR(45) | - | NULL | 客户端IP |
| user_id | VARCHAR(36) | INDEX | NULL | 操作用户ID |
| error_code | VARCHAR(50) | INDEX | NULL | 错误代码 |
| error_message | TEXT | - | NULL | 错误信息 |
| retry_count | INT | NOT NULL | 0 | 重试次数 |
| created_at | DATETIME | NOT NULL | CURRENT_TIMESTAMP | 记录时间 |

**索引设计**：
- 主键索引：`PRIMARY KEY (id)`
- 唯一索引：`UNIQUE KEY uk_request_id (request_id)`
- 查询索引：`INDEX idx_caller_callee (caller_module, callee_module, start_time DESC)`, `INDEX idx_response_status (response_status)`, `INDEX idx_duration (duration_ms)` 用于慢查询分析
- 时间索引：`INDEX idx_start_time (start_time)` 按时间范围查询

**业务规则**：
1. 请求ID贯穿调用链，用于分布式追踪
2. 敏感信息（密码、token）在存储前脱敏
3. 响应体只存储前1KB，完整日志存日志系统
4. 慢查询阈值：duration_ms > 5000ms 标记为慢查询
5. 错误请求：response_status >= 400 标记为错误

### 2.4 消息队列状态表 (mq_status_monitor)
监控各模块消息队列消费状态，保障事件驱动架构可靠性。

| 字段名 | 数据类型 | 约束 | 默认值 | 说明 |
|--------|----------|------|--------|------|
| id | BIGINT | PRIMARY KEY AUTO_INCREMENT | - | 自增主键 |
| module_name | VARCHAR(30) | NOT NULL INDEX | - | 模块名称 |
| queue_name | VARCHAR(100) | NOT NULL INDEX | - | 队列名称 |
| queue_type | ENUM('event','command','notification','dead_letter') | NOT NULL | 'event' | 队列类型 |
| consumer_group | VARCHAR(50) | NOT NULL INDEX | - | 消费组 |
| partition | INT | NOT NULL | 0 | 分区编号 |
| total_messages | BIGINT | NOT NULL | 0 | 队列总消息数 |
| pending_messages | BIGINT | NOT NULL | 0 | 待消费消息数 |
| processing_messages | BIGINT | NOT NULL | 0 | 处理中消息数 |
| last_consumed_offset | BIGINT | - | NULL | 最后消费偏移量 |
| last_consumed_time | DATETIME(3) | INDEX | NULL | 最后消费时间 |
| consumer_lag | BIGINT | NOT NULL | 0 | 消费延迟（消息数） |
| consumer_status | ENUM('active','inactive','lagging','error') | NOT NULL INDEX | 'active' | 消费状态 |
| error_count | INT | NOT NULL | 0 | 错误计数 |
| last_error_time | DATETIME | - | NULL | 最后错误时间 |
| last_error_message | TEXT | - | NULL | 最后错误信息 |
| collected_at | DATETIME(3) | NOT NULL INDEX | CURRENT_TIMESTAMP(3) | 监控数据采集时间 |
| created_at | DATETIME | NOT NULL | CURRENT_TIMESTAMP | 记录创建时间 |
| updated_at | DATETIME | NOT NULL | CURRENT_TIMESTAMP ON UPDATE | 更新时间 |

**索引设计**：
- 主键索引：`PRIMARY KEY (id)`
- 唯一索引：`UNIQUE KEY uk_module_queue_group (module_name, queue_name, consumer_group, partition)`
- 监控索引：`INDEX idx_status_collected (consumer_status, collected_at)` 用于实时监控
- 延迟索引：`INDEX idx_lag_collected (consumer_lag DESC, collected_at)` 识别高延迟队列

**业务规则**：
1. 每30秒采集一次队列状态
2. consumer_lag > 1000 标记为 lagging 状态
3. last_consumed_time超过5分钟无更新标记为 inactive
4. error_count > 10 触发告警

4. last_check_time超过2倍检查间隔未更新标记为 unknown

### 3.2 关键数据流说明

#### 3.2.1 设备监控 → 报警管理 (实时事件流)
- **触发条件**：设备状态变更、指标超阈值
- **数据载体**：集成事件表 + 消息队列
- **传输协议**：MQTT/AMQP + JSON Schema
- **SLA要求**：端到端延迟 < 1秒，可靠性 > 99.9%

#### 3.2.2 数据采集 → 报表统计 (批量同步流)
- **触发条件**：定时任务（每小时、每天）
- **数据载体**：数据同步表 + 文件传输
- **传输协议**：SFTP/HTTP + Parquet格式
- **SLA要求**：数据延迟 < 5分钟，完整性100%

#### 3.2.3 用户权限 → 所有模块 (实时同步流)
- **触发条件**：用户/角色/权限变更
- **数据载体**：共享数据库 + 缓存失效
- **传输协议**：数据库事务 + Redis Pub/Sub
- **SLA要求**：变更生效延迟 < 100ms，一致性最终一致

#### 3.2.4 系统配置 → 所有模块 (配置推送流)
- **触发条件**：配置项变更
- **数据载体**：配置中心 + 长连接推送
- **传输协议**：gRPC + Protobuf
- **SLA要求**：配置生效延迟 < 30秒，版本一致性

```json
  "payload": {
    "before": {"status": "online", "health_score": 85},
    "after": {"status": "offline", "health_score": 70},
    "changed_fields": ["status", "health_score"]
  },
  "metadata": {
    "triggered_by": "user_123",
    "correlation_id": "req_789",
    "trace_id": "trace_456"
  }
}
```

#### 4.1.2 事件处理流程
1. **事件发布**：源模块发布事件到事件总线
2. **事件路由**：根据事件类型路由到对应队列
3. **事件消费**：消费者模块订阅并处理事件
4. **事件确认**：成功处理后发送ACK
5. **事件归档**：处理完成的事件移动到归档存储

#### 4.1.3 容错机制
- **重试策略**：指数退避重试，最多3次
- **死信队列**：处理失败事件进入死信队列人工处理
- **幂等处理**：基于event_id实现消费幂等性
- **补偿事务**：失败时执行补偿操作保证一致性

## 5. 数据一致性保障

### 5.1 一致性级别定义
| 级别 | 描述 | 适用场景 | 实现机制 |
|------|------|----------|----------|
| 强一致 | 所有模块实时一致 | 用户权限、配置数据 | 分布式事务、同步调用 |
| 最终一致 | 一定延迟后一致 | 设备数据、报表数据 | 事件驱动、异步同步 |
| 弱一致 | 不保证一致性 | 监控指标、日志数据 | 无协调、定期清理 |

### 5.2 分布式事务方案

#### 5.2.1 Saga模式（长事务）
```yaml
- 步骤1: 创建设备（设备模块）
- 步骤2: 初始化指标（采集模块）
- 步骤3: 创建默认阈值（报警模块）
- 补偿1: 删除设备（失败时）
- 补偿2: 清理指标数据（失败时）
- 补偿3: 删除阈值配置（失败时）
```

#### 5.2.2 TCC模式（短事务）
- **Try阶段**：预留资源，检查约束
- **Confirm阶段**：确认操作，提交事务
- **Cancel阶段**：取消操作，释放资源

### 5.3 数据验证机制
1. **格式验证**：JSON Schema验证数据格式
2. **业务验证**：业务规则引擎验证逻辑
3. **引用验证**：外键关系验证数据引用
4. **历史对比**：与历史数据对比检测异常
