## 1. 设计概述

### 1.1 业务领域
设备监控模块是NESOM系统的核心，负责新能源设备的全生命周期监控和管理。主要业务领域包括：

1. **设备资产管理**：设备档案、分类、属性、位置信息管理
2. **实时监控**：设备运行状态、性能指标实时采集和展示
3. **异常检测**：阈值告警、故障诊断、预警预测
4. **维护管理**：预防性维护、维修记录、健康评估
5. **数据分析**：运行统计、性能分析、能效评估

### 1.2 核心业务流程
1. 设备接入 → 数据采集 → 状态监控 → 异常告警 → 工单处理 → 维护记录
2. 设备台账 → 维护计划 → 任务执行 → 结果记录 → 健康评估
3. 数据采集 → 存储归档 → 统计分析 → 报表生成 → 决策支持

### 1.3 设计原则
1. **实时性优先**：监控数据秒级更新，告警分钟级响应
2. **可靠性保障**：数据不丢失，故障自恢复，服务高可用
3. **可扩展性**：支持多协议、多设备类型、大数据量
4. **智能化**：趋势预测、故障预警、智能诊断
5. **安全性**：操作可追溯，数据可审计，权限精细化

## 2. 核心业务实体状态机

### 2.1 设备状态机 (Device Status State Machine)

#### 2.1.1 状态定义
| 状态 | 编码 | 描述 | 业务含义 |
|------|------|------|----------|
| 离线 | offline | 设备未连接或通讯中断 | 设备无法采集数据 |
| 在线 | online | 设备正常连接并运行 | 数据正常采集，可远程控制 |
| 故障 | fault | 设备检测到硬件或软件故障 | 需立即维修，可能影响生产 |
| 维护中 | maintenance | 设备正在进行维护 | 暂停数据采集，不可远程控制 |
| 测试中 | testing | 设备处于测试模式 | 数据采集但不计入统计 |
| 备用 | standby | 设备处于备用状态 | 设备正常，但未投入运行 |

#### 2.1.2 状态转换规则
```
                            ┌─────────────┐
                            │   offline   │◀──────────────────┐
                            └─────────────┘                   │
                                   │                          │
        心跳恢复/手动上线          │ 心跳丢失/手动下线        │
                                   ▼                          │
                            ┌─────────────┐                  │
    ┌───────────────────────│   online    │──────────────────┘
    │                       └─────────────┘
    │                              │
    │检测到故障/告警触发            │ 手动开始维护
    │                              ▼
    │                       ┌─────────────┐
    └──────────────────────▶│ maintenance │
    │                       └─────────────┘
    │                              │
    │维护完成/故障修复              │ 手动开始测试
    │                              ▼
    │                       ┌─────────────┐
    └──────────────────────▶│   testing   │
                            └─────────────┘
                                   │
                                   │ 测试完成
                                   ▼
                            ┌─────────────┐
                            │   standby   │
                            └─────────────┘
```

#### 2.1.3 状态转换约束表
| 当前状态 | 目标状态 | 触发条件 | 前置检查 | 后置动作 |
|----------|----------|----------|----------|----------|
| offline | online | 1. 心跳恢复<br>2. 手动上线操作 | 1. 设备类型有效<br>2. 数据采集配置正确 | 1. 开始数据采集<br>2. 记录上线事件<br>3. 通知相关人员 |
| online | offline | 1. 心跳丢失(>5分钟)<br>2. 手动下线操作 | 1. 无正在执行的控制指令 | 1. 停止数据采集<br>2. 记录下线事件<br>3. 发送离线通知 |
| online | fault | 1. 检测到严重故障<br>2. 连续告警达到阈值 | 1. 故障诊断验证 | 1. 停止数据采集<br>2. 生成故障工单<br>3. 发送紧急通知 |
| online | maintenance | 1. 手动开始维护 | 1. 无正在执行的控制指令<br>2. 维护计划已审批 | 1. 停止数据采集<br>2. 生成维护记录<br>3. 通知相关人员 |
| maintenance | online | 1. 维护完成确认 | 1. 维护记录已填写<br>2. 设备测试通过 | 1. 恢复数据采集<br>2. 更新维护计划<br>3. 重新计算健康评分 |
| any | testing | 1. 手动开始测试 | 1. 设备不在故障状态 | 1. 切换到测试数据采集<br>2. 记录测试开始事件 |
| testing | standby | 1. 测试完成确认 | 1. 测试报告已提交 | 1. 停止测试数据采集<br>2. 记录测试结果<br>3. 评估设备性能 |

#### 2.1.4 状态转换业务规则
1. **自动转换规则**：
   - 心跳丢失超过5分钟：online → offline（自动）
   - 连续3个critical告警：online → fault（自动）
   - 维护计划到期：online → maintenance（需确认）

2. **人工确认规则**：
   - fault → maintenance：需主管审批
   - maintenance → online：需测试报告
   - testing → standby：需质量确认

3. **禁止转换规则**：
   - fault状态不能直接转为online
   - 有未完成工单时不能转为maintenance以外的状态

### 2.2 告警状态机 (Alert Status State Machine)

#### 2.2.1 状态定义
| 状态 | 编码 | 描述 | 业务含义 |
|------|------|------|----------|
| 活跃 | active | 告警已产生，未处理 | 需要关注和处理 |
| 已确认 | acknowledged | 告警已被人员确认 | 已关注，正在处理中 |
| 已解决 | resolved | 告警问题已解决 | 问题已处理，待关闭 |
| 已关闭 | closed | 告警流程结束 | 问题完全解决，流程结束 |

#### 2.2.2 状态转换规则
```
     告警产生
         │
         ▼
   ┌─────────────┐
   │   active    │
   └─────────────┘
         │
         │ 人工确认
         ▼
   ┌─────────────┐
   │acknowledged │
   └─────────────┘
         │
         │ 问题解决
         ▼
   ┌─────────────┐
   │  resolved   │
   └─────────────┘
         │
         │ 关闭确认
         ▼
   ┌─────────────┐
   │   closed    │
   └─────────────┘
```

#### 2.2.3 状态转换约束
| 当前状态 | 目标状态 | 触发条件 | 前置检查 | 后置动作 |
|----------|----------|----------|----------|----------|
| active | acknowledged | 1. 人工确认操作 | 1. 告警未过期<br>2. 用户有确认权限 | 1. 记录确认人和时间<br>2. 发送确认通知 |
| active | resolved | 1. 自动恢复(阈值)<br>2. 手动解决 | 1. 问题已实际解决 | 1. 填写解决说明<br>2. 可选创建工单<br>3. 发送解决通知 |
| acknowledged | resolved | 1. 手动解决 | 1. 填写解决说明 | 1. 更新解决信息<br>2. 关闭相关通知 |
| resolved | closed | 1. 人工关闭<br>2. 自动关闭(7天后) | 1. 无关联活动工单 | 1. 记录关闭人和时间<br>2. 归档告警记录 |

#### 2.2.4 告警升级规则
1. **时间升级**：
   - active状态超过1小时 → 通知主管
   - active状态超过4小时 → 通知部门经理
   - active状态超过24小时 → 通知值班领导

2. **级别升级**：
   - warning告警连续出现3次 → 升级为error
   - error告警1小时内未确认 → 升级为critical

3. **渠道升级**：
   - warning: 站内消息
   - error: 站内消息 + 短信
   - critical: 站内消息 + 短信 + 电话

### 2.3 维护记录状态机 (Maintenance Status State Machine)

#### 2.3.1 状态定义
| 状态 | 编码 | 描述 | 业务含义 |
|------|------|------|----------|
| 计划中 | scheduled | 维护计划已制定 | 等待执行 |
| 进行中 | in_progress | 维护任务已开始 | 正在执行维护 |
| 已完成 | completed | 维护任务已完成 | 维护完成，待验收 |
| 已取消 | cancelled | 维护任务已取消 | 计划取消 |

#### 2.3.2 状态转换规则
```
     计划创建
         │
         ▼
   ┌─────────────┐
   │  scheduled  │
   └─────────────┘
         │
         │ 开始维护
         ▼
   ┌─────────────┐
   │ in_progress │
   └─────────────┘
         │
         │ 完成维护
         ▼
   ┌─────────────┐
   │  completed  │
   └─────────────┘
```

#### 2.3.3 状态转换业务规则
1. **计划创建规则**：
   - 基于维护模板自动生成
   - 支持手动创建临时维护
   - 支持从告警自动创建

2. **开始维护规则**：
   - 设备必须为online或offline状态
   - 维护人员必须有相应资质
   - 需要安全审批（高危作业）

3. **完成维护规则**：
   - 必须填写维护报告
   - 必须上传现场照片
   - 必须记录使用备件

## 3. 核心业务流程设计

### 3.1 设备接入流程

#### 3.1.1 正常路径
**触发**: 新设备到货或需要接入系统

**步骤**:
1. **STEP 1: 设备信息登记**
   - 输入设备基础信息（编码、名称、类型、规格）
   - 选择所属场站和位置
   - 上传设备资料（说明书、合格证、图片）

2. **STEP 2: 采集配置**
   - 选择通信协议（Modbus TCP/RTU, OPC UA, MQTT等）
   - 配置采集参数（地址、端口、寄存器映射）
   - 设置采集频率和告警阈值

3. **STEP 3: 设备调试**
   - 系统尝试连接设备
   - 验证数据采集准确性
   - 测试控制指令（如启停、参数设置）

4. **STEP 4: 正式上线**
   - 标记设备为online状态
   - 开始正常数据采集
   - 纳入监控和统计范围

#### 3.1.2 异常分支
- **设备连接失败**：检查网络、配置、设备状态
- **数据采集异常**：验证寄存器映射、数据类型
- **权限验证失败**：检查设备访问权限
- **重复设备编码**：修改设备编码或合并记录

#### 3.1.3 业务规则
1. 设备编码必须唯一（场站内）
2. 关键设备需双重验证（技术员+主管）
3. 调试期间数据不计入统计
4. 上线前必须完成基础测试

### 3.2 实时监控流程

#### 3.2.1 数据采集流程
**触发**: 定时任务（秒级/分钟级）

**步骤**:
1. **STEP 1: 采集调度**
   - 查询需要采集的设备列表
   - 根据优先级分配采集任务
   - 负载均衡（避免集中采集）

2. **STEP 2: 数据获取**
   - 建立设备连接（TCP/串口/MQTT）
   - 读取寄存器/点位数据
   - 记录采集时间和质量

3. **STEP 3: 数据解析**
   - 原始数据解析（字节转数值）
   - 单位换算和校准
   - 数据质量评估

4. **STEP 4: 数据存储**
   - 实时数据存入缓存（Redis）
   - 历史数据批量入库（MySQL）
   - 触发告警检查

#### 3.2.2 性能优化策略
1. **连接池管理**：复用设备连接，减少握手开销
2. **批量读取**：一次读取多个寄存器，减少通信次数
3. **异步处理**：采集与处理分离，提高吞吐量
4. **故障转移**：主备采集器，自动切换

#### 3.2.3 数据质量保障
1. **完整性检查**：必填字段不能为空
2. **合理性检查**：数值在合理范围内
3. **连续性检查**：数据变化不能突变
4. **一致性检查**：相关数据逻辑一致

### 3.3 告警处理流程

#### 3.3.1 告警产生流程
**触发**: 数据采集后或定时检查

**步骤**:
1. **STEP 1: 阈值检查**
   - 获取设备当前指标值
   - 查询设备阈值配置（设备级→类型级→全局级）
   - 比较数值与阈值

2. **STEP 2: 告警判定**
   - 判断告警级别（warning/error/critical）
   - 检查是否重复告警（相同设备相同类型）
   - 验证告警有效性（避免误报）

3. **STEP 3: 告警生成**
   - 生成唯一告警编码
   - 记录告警详情（设备、指标、数值、阈值）
   - 设置告警状态为active

4. **STEP 4: 通知发送**
   - 根据告警级别选择通知方式
   - 获取通知接收人（设备负责人、值班人员）
   - 发送通知并记录状态

#### 3.3.2 告警抑制规则
1. **重复告警抑制**：相同告警5分钟内不重复产生
2. **维护期抑制**：维护期间只产生info级别告警
3. **测试期抑制**：测试数据不触发生产告警
4. **时间段抑制**：非工作时间降低告警级别

#### 3.3.3 智能告警特性
1. **趋势预警**：基于历史趋势预测可能告警
2. **关联分析**：关联设备告警综合分析
3. **根因分析**：基于设备拓扑分析根本原因
4. **自动恢复**：临时波动告警自动恢复

### 3.4 维护管理流程

#### 3.4.1 预防性维护流程
**触发**: 维护计划到期或设备运行时间达到

**步骤**:
1. **STEP 1: 计划生成**
   - 基于维护模板生成维护计划
   - 指定维护人员和预计时长
   - 设置维护窗口（避免影响生产）

2. **STEP 2: 任务分配**
   - 分配具体维护任务给技术人员
   - 发送任务通知（移动端）
   - 确认人员资质和可用性

3. **STEP 3: 现场执行**
   - 现场扫码确认设备
   - 执行维护检查项
   - 记录检查结果和发现

4. **STEP 4: 结果上报**
   - 填写维护报告
   - 上传现场照片
   - 记录使用备件

5. **STEP 5: 验收关闭**
   - 主管验收维护质量
   - 更新设备维护记录
   - 调整下次维护计划

#### 3.4.2 纠正性维护流程
**触发**: 设备故障或性能下降

**步骤**:
1. **STEP 1: 故障诊断**
   - 分析告警和运行数据
   - 现场检查设备状态
   - 确定故障原因和范围

2. **STEP 2: 维修方案**
   - 制定维修方案和备件需求
   - 评估维修时间和影响
   - 审批维修计划

3. **STEP 3: 维修执行**
   - 设备隔离和安全措施
   - 更换故障部件
   - 功能测试和验证

4. **STEP 4: 恢复运行**
   - 设备重新上线
   - 监控运行状态
   - 更新故障库

## 4. 业务规则详细设计

### 4.1 设备编码规则
**规则ID**: DEV-CODE-001  
**描述**: 设备编码生成规则  
**公式**: `{场站代码}-{设备类型代码}-{序号}`  
**示例**: `SUN-PV-PVINV-001`  
**约束**:
1. 场站代码: 3-8位字母数字
2. 设备类型代码: 2-10位字母
3. 序号: 3位数字，场站内自增
4. 总长度不超过50字符

**验证逻辑**:
```python
def validate_device_code(station_code, device_type_code, sequence):
    if not re.match(r'^[A-Z0-9]{3,8}$', station_code):
        return False, "场站代码格式错误"
    if not re.match(r'^[A-Z]{2,10}$', device_type_code):
        return False, "设备类型代码格式错误"
    if not 1 <= sequence <= 999:
        return False, "序号超出范围"
    return True, f"{station_code}-{device_type_code}-{sequence:03d}"
```

### 4.2 健康评分计算规则
**规则ID**: HEALTH-SCORE-001  
**描述**: 设备健康评分计算  
**公式**: `健康评分 = 基础分 - 故障扣分 - 性能扣分 + 维护加分`

**计算要素**:
1. **基础分**: 100分（全新设备）
2. **故障扣分**: 
   - critical告警: -10分/次
   - error告警: -5分/次  
   - warning告警: -2分/次
3. **性能扣分**:
   - 效率低于标准: -（差值×10）分
   - 连续运行超时: -1分/1000小时
4. **维护加分**:
   - 按时维护: +5分/次
   - 预防性维护: +3分/次

**更新时机**:
1. 每日凌晨自动计算
2. 告警产生时实时更新
3. 维护完成后立即更新

### 4.3 告警阈值设置规则
**规则ID**: THRESHOLD-001  
**描述**: 多级阈值设置规则

**阈值级别**:
1. **警告阈值** (warning): 正常范围边界，需要关注
2. **错误阈值** (error): 异常范围，需要处理
3. **严重阈值** (critical): 危险范围，立即处理

**设置原则**:
1. **设备级优先**: 设备特定阈值 > 类型级阈值 > 全局阈值
2. **季节调整**: 夏季温度阈值高于冬季
3. **负载相关**: 高负载时电流阈值可适当放宽
4. **时间衰减**: 阈值随时间自动调整（学习机制）

### 4.4 数据采集频率规则
**规则ID**: COLLECT-FREQ-001  
**描述**: 不同数据类型采集频率

**频率表**:
| 数据类型 | 正常频率 | 告警时频率 | 维护时频率 |
|----------|----------|------------|------------|
| 电压/电流 | 1分钟 | 10秒 | 暂停 |

## 5. 异常处理机制

### 5.1 数据采集异常

#### 5.1.1 通信超时处理
**异常类型**: 设备无响应或响应超时  
**处理流程**:
1. **重试策略**: 最多重试3次，间隔1秒、3秒、5秒
2. **状态更新**: 连续3次失败标记设备为offline
3. **告警产生**: 产生communication_error告警
4. **恢复检测**: 每5分钟尝试恢复连接

#### 5.1.2 数据解析异常
**异常类型**: 数据格式错误或校验失败  
**处理流程**:
1. **数据丢弃**: 标记为无效数据，不计入统计
2. **质量标记**: 数据质量标记为0
3. **日志记录**: 记录详细错误信息供分析
4. **配置检查**: 提示检查采集配置

#### 5.1.3 数据库异常
**异常类型**: 数据存储失败  
**处理流程**:
1. **缓存暂存**: 数据暂存Redis，设置TTL
2. **异步重试**: 后台任务定时重试存储
3. **降级处理**: 关键数据优先，非关键可丢弃
4. **监控告警**: 存储失败率超阈值告警

### 5.2 业务逻辑异常

#### 5.2.1 状态转换冲突
**异常类型**: 非法状态转换  
**处理流程**:
1. **操作拒绝**: 返回详细错误信息
2. **状态验证**: 提供当前状态和允许操作
3. **审批流程**: 特殊转换需走审批流程
4. **审计记录**: 记录非法操作尝试

#### 5.2.2 权限不足
**异常类型**: 用户无操作权限  
**处理流程**:
1. **操作拦截**: API层统一拦截
2. **权限提示**: 明确提示所需权限
3. **申请入口**: 提供权限申请链接
4. **管理员通知**: 记录权限尝试日志

#### 5.2.3 数据一致性异常
**异常类型**: 数据版本冲突或引用不存在  
**处理流程**:
1. **乐观锁**: 使用版本号防止并发修改
2. **事务回滚**: 相关操作在一个事务中
3. **引用检查**: 操作前验证外键引用
4. **补偿机制**: 失败操作的补偿处理

### 5.3 系统级异常

#### 5.3.1 服务不可用
**异常类型**: 依赖服务（数据库、Redis）不可用  
**处理流程**:
1. **健康检查**: 定期检查依赖服务状态
2. **熔断机制**: 失败率超阈值自动熔断
3. **降级策略**: 返回缓存数据或默认值
4. **快速失败**: 避免长时间等待

#### 5.3.2 资源耗尽
**异常类型**: 内存、连接数等资源不足  
**处理流程**:
1. **资源监控**: 实时监控资源使用率
2. **限流保护**: 请求限流，保护核心功能
3. **优雅降级**: 关闭非核心功能
4. **自动扩容**: 云环境自动扩容

### 6.2 业务数据验证

#### 6.2.1 状态转换验证
**规则**: 验证状态转换的合法性  
**实现**:
```python
def validate_status_transition(current_status, target_status, device, user):
    # 定义允许的转换
    allowed_transitions = {
        "offline": ["online", "maintenance"],
        "online": ["offline", "fault", "maintenance", "testing"],
        "fault": ["maintenance"],
        "maintenance": ["online", "testing"],
        "testing": ["standby", "online"],
        "standby": ["online", "offline"]
    }
    
    # 检查基本转换
    if target_status not in allowed_transitions.get(current_status, []):
        return False, f"不允许从{current_status}转换到{target_status}"
    
    # 业务规则检查
    if current_status == "online" and target_status == "maintenance":
        if device.has_active_alerts():
            return False, "设备有活跃告警，不能进入维护"
        if not user.has_permission("device.maintenance"):
            return False, "用户无维护权限"
    
    return True, "转换验证通过"
```

#### 6.2.2 告警确认验证
**规则**: 验证告警确认的合法性  
**实现**:
```python
def validate_alert_acknowledge(alert, user):
    # 状态检查
    if alert.status != "active":
        return False, "只有活跃告警可以确认"
    
    # 时间检查（告警产生超过24小时不能确认）
    if (datetime.now() - alert.start_time).total_seconds() > 86400:
        return False, "告警超过24小时，请先升级处理"
    
    # 权限检查
    if not user.has_permission(f"alert.acknowledge.{alert.alert_level}"):
        return False, f"用户无权确认{alert.alert_level}级别告警"
    
    # 重复确认检查
    if alert.acknowledged_by == user.id:
        return False, "用户已确认过此告警"
    
    return True, "确认验证通过"
```

   - 历史数据: (device_id, collected_at) DESC

   - 3年前数据移至历史库
   - 冷数据压缩存储
   - 统计表预计算

#### 7.1.2 写入优化
1. **批量写入**: 历史数据批量提交，减少事务开销
2. **异步写入**: 非关键数据异步存储
3. **缓存写入**: 先写缓存，批量落库
4. **分库分表**: 数据量大时按场站分库

### 7.2 缓存策略

#### 7.2.1 缓存层级
1. **L1缓存** (内存): 
   - 设备实时数据 (5秒TTL)
   - 设备状态 (30秒TTL)
   - 活跃告警 (60秒TTL)

2. **L2缓存** (Redis):
   - 设备详情 (5分钟TTL)
   - 设备列表查询结果 (2分钟TTL)
   - 历史数据聚合结果 (10分钟TTL)

3. **L3缓存** (CDN):
   - 设备静态资源（图片、文档）
   - 报表文件
   - 配置模板

#### 7.2.2 缓存更新策略
1. **主动更新**: 数据变更时主动更新缓存
2. **被动更新**: 缓存失效时重新加载
3. **预加载**: 热点数据提前加载
4. **写穿透**: 写操作同时更新缓存
