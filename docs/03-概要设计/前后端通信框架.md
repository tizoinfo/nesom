## 1. 概述

本文档定义NESOM系统前后端通信的完整规范，包括API设计原则、数据格式标准、错误处理机制、认证授权方案和版本管理策略。采用"契约先行"的设计方法，确保前后端并行开发。

### 1.1 设计目标
1. **一致性**：统一的API设计风格和响应格式
2. **可读性**：清晰的接口文档和示例
3. **安全性**：完善的认证授权机制
4. **可维护性**：版本管理，向后兼容
5. **可测试性**：易于自动化测试

### 1.2 设计原则
1. **RESTful风格**：资源导向，使用HTTP语义
2. **无状态性**：服务端不保存客户端状态
3. **自描述消息**：请求响应包含足够信息
4. **超媒体驱动**：响应包含相关资源链接（HATEOAS）

## 2. API设计规范

### 2.1 URL规范

#### 2.1.1 格式要求
```
协议://域名[:端口]/api/版本/模块/资源[/标识符[/子资源]]
```

#### 2.1.2 示例
```
# 获取设备列表
GET https://api.nesom.com/api/v1/devices

# 获取单个设备
GET https://api.nesom.com/api/v1/devices/123

# 获取设备的告警列表
GET https://api.nesom.com/api/v1/devices/123/alerts

# 创建设备
POST https://api.nesom.com/api/v1/devices

# 更新设备
PUT https://api.nesom.com/api/v1/devices/123

# 删除设备
DELETE https://api.nesom.com/api/v1/devices/123
```

#### 2.1.3 命名规则
- **资源名**：使用复数名词，小写，单词间用连字符 `-`
- **操作名**：使用动词，如 `/devices/123/start`
- **查询参数**：小写，单词间用下划线 `_`

### 2.2 HTTP方法使用

| 方法 | 语义 | 幂等性 | 安全性 | 示例 |
|------|------|--------|--------|------|
| GET | 获取资源 | 是 | 是 | `GET /devices` |
| POST | 创建资源 | 否 | 否 | `POST /devices` |
| PUT | 更新资源（全量） | 是 | 否 | `PUT /devices/123` |
| PATCH | 更新资源（部分） | 否 | 否 | `PATCH /devices/123` |
| DELETE | 删除资源 | 是 | 否 | `DELETE /devices/123` |
| HEAD | 获取头部信息 | 是 | 是 | `HEAD /devices` |
| OPTIONS | 获取支持的方法 | 是 | 是 | `OPTIONS /devices` |

## 5. 认证授权方案

### 5.1 JWT认证流程

```
┌─────────┐    1.登录请求     ┌─────────┐    2.验证凭证     ┌─────────┐
│  客户端  ├─────────────────►│  认证服务 ├─────────────────►│  数据库  │
│         │                  │         │                  │         │
│         ◄──────────────────┤         ◄──────────────────┤         │
│         │    3.返回JWT     │         │    4.用户信息    │         │
└─────────┘                  └─────────┘                  └─────────┘
       │                            │
       │ 5.携带JWT访问API           │
       └───────────────────────────►└─────────┐
                                              │
                                      6.验证JWT │
                                              ▼
                                        ┌─────────┐
                                        │ 业务API │
                                        └─────────┘
```

### 5.2 Token结构

#### 5.2.1 JWT Payload示例
```json
{
  "sub": "1234567890",        // 用户ID
  "username": "zhangsan",     // 用户名
  "roles": ["technician", "inspector"],  // 角色列表
  "permissions": [            // 权限列表
    "device:read",
    "workorder:create", 
    "inspection:update"
  ],
  "iat": 1516239022,          // 签发时间
  "exp": 1516242622,          // 过期时间 (1小时后)
  "iss": "nesom-auth",        // 签发者
  "aud": "nesom-api"          // 接收者
}
```

#### 5.2.2 Token管理
- **Access Token**：短期令牌，15分钟过期，用于API访问
- **Refresh Token**：长期令牌，7天过期，用于刷新Access Token
- **Token存储**：前端存储localStorage或HttpOnly Cookie
- **Token刷新**：通过Refresh Token获取新的Access Token

### 5.3 权限控制

#### 5.3.1 RBAC模型
```
用户 (User) → 角色 (Role) → 权限 (Permission) → 资源 (Resource)
```

#### 5.3.2 权限定义格式
```
{资源}:{操作}:{范围}
```

#### 5.3.3 权限示例
```json
{
  "device:read:*",           // 可读所有设备
  "device:write:own",        // 可写自己的设备
  "workorder:create:*",      // 可创建所有工单
  "workorder:delete:assigned", // 可删除分配的工单
  "report:export:*",         // 可导出所有报表
  "system:config:read"       // 可读系统配置
}
```

### 5.4 接口保护

#### 5.4.1 认证中间件
```python
# FastAPI中间件示例
async def auth_middleware(request: Request, call_next):
    # 1. 从Header提取Token
    token = request.headers.get("Authorization")
    if not token or not token.startswith("Bearer "):
        raise HTTPException(status_code=401, detail="未提供认证令牌")
    
    token = token[7:]  # 移除"Bearer "
    
    # 2. 验证Token
    try:
        payload = jwt.decode(
            token, 
            settings.SECRET_KEY, 
            algorithms=[settings.ALGORITHM],
            issuer="nesom-auth",
            audience="nesom-api"
        )
        request.state.user = payload
    except jwt.ExpiredSignatureError:
        raise HTTPException(status_code=401, detail="令牌已过期")
    except jwt.InvalidTokenError:
        raise HTTPException(status_code=401, detail="无效的令牌")
    
    # 3. 继续处理请求
    response = await call_next(request)
    return response
```

#### 5.4.2 权限装饰器
```python
def require_permission(permission: str):
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, **kwargs):
            request = kwargs.get("request")
            if not request:
                raise HTTPException(status_code=500, detail="请求对象未找到")
            
            user_permissions = request.state.user.get("permissions", [])
            if permission not in user_permissions:
                raise HTTPException(status_code=403, detail="权限不足")
            
            return await func(*args, **kwargs)
        return wrapper
    return decorator

# 使用示例
@router.get("/devices/{id}")
@require_permission("device:read:*")
async def get_device(id: int, request: Request):
    # 有权限才执行
    pass
```

## 6. 版本管理策略

### 6.1 版本号格式
```
v{major}.{minor}.{patch}
```

### 6.2 版本升级规则

| 版本变化 | 说明 | 兼容性要求 |
|----------|------|------------|
| 主版本(major) | 不兼容的API变更 | 需要客户端升级 |
| 次版本(minor) | 向后兼容的功能性新增 | 客户端可选升级 |
| 修订版本(patch) | 向后兼容的问题修复 | 客户端透明 |

### 6.3 URL版本控制

#### 6.3.1 URL路径版本
```
/api/v1/devices    # 版本在URL路径中
```

#### 6.3.2 Header版本
```http
GET /api/devices
Accept: application/vnd.nesom.v1+json
```

#### 6.3.3 默认方案：URL路径版本
- 简单直观
- 易于调试和文档化
- 支持多版本共存

### 6.4 版本迁移策略

#### 6.4.1 并行支持
```
v1 → v2 迁移期间：
- 同时支持 /api/v1/* 和 /api/v2/*
- 新功能只在v2添加
- 逐步迁移客户端到v2
- v1维护至少6个月
```

#### 6.4.2 废弃通知
```http
GET /api/v1/devices
{
  "code": 200,
  "message": "success",
  "data": [...],
  "meta": {
    "deprecation_warning": "此API版本(v1)已废弃，将于2026-10-01停用，请迁移到v2"
  }
}
```
