## 文档贡献指南

本仓库当前以 `docs/` 为中心，目标是让设计与决策可追溯、可评审、可执行。

## 1) 放置规则（去哪写）

- **总体结构**：优先按 `docs/01-...` ~ `docs/08-...` 分类存放
- **详细设计**：放在 `docs/06-详细设计/`，再按模块或专题拆分
- **模板与评审材料**：优先复用 `docs/06-详细设计/设计评审/` 与 `docs/06-详细设计/评审/` 下的模板

## 2) 命名规范（怎么命名）

- **推荐格式**：`YYYY-MM-DD-文档标题.md`
  - 例：`2026-04-09-设备监控模块接口设计.md`
- **不带日期的固定入口文档**：允许使用稳定文件名
  - 例：`README.md`、`INDEX.md`、`模块评审摘要.md`
- **评审归档版本**：使用后缀 `*_reviewed.md`
  - 例：`API接口设计_reviewed.md`

## 3) 文档头部元信息（必须）

建议所有“可评审文档”在头部包含以下字段（可复制粘贴后填写）：

```
版本: 1.0
日期: YYYY-MM-DD
作者: <姓名/角色>
状态: Draft | Reviewed
审核状态: 待评审 | 需修改 | 已通过
```

说明：
- **Draft**：内容可能变化，允许快速迭代
- **Reviewed**：代表已走完评审流程并归档

## 4) 评审流程（怎么评）

- **提交评审前**：
  - 确保文档目标、范围、关键决策点清晰
  - 在文档中给出需要评审者确认的问题清单（如架构选择、边界、风险）
- **评审中**：
  - 使用现有评审模板记录问题与结论
  - 对关键决策形成可追溯记录（建议在文档“决策记录/变更记录”章节追加）
- **评审后**：
  - 若通过：将 `审核状态` 更新为“已通过”，必要时生成 `*_reviewed.md` 归档版
  - 若需修改：将 `审核状态` 更新为“需修改”，并在文档内记录修改点与计划

## 5) 变更记录（可追溯性）

对重要文档，建议在末尾维护“更新记录”：

```
## 更新记录
- YYYY-MM-DD：<一句话说明本次变更>
```

## 6) 文档入口维护（保持可发现）

新增或调整关键文档后，请同步更新：

- `docs/INDEX.md`（推荐阅读路径与链接）
- 如涉及目录层级变化：`docs/README.md`
